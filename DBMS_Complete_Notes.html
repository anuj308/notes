01 Introduction to DBMS

▼

📌 Quick Summary

DBMS is software to store, manage & retrieve data efficiently. It overcomes file system limitations like redundancy, inconsistency, security issues.

What is Data?

Data is a collection of raw, unorganized facts (text, observations, figures, symbols). It has no meaning by itself until processed.

Types of Data

📊 Quantitative
Numerical form — weight, volume, cost of an item.
📝 Qualitative
Descriptive, not numerical — name, gender, hair color.

What is Information?

Processed, organized, structured data that provides context and enables decision making. E.g., analyzing locality data to conclude "there are 100 senior citizens" — that's information.

Data vs Information

DataInformation
Raw, unorganized factsOrganized, processed data
No meaning by itselfHas context & meaning
Doesn't depend on informationDepends on data
Not sufficient for decision-makingEnables decision-making
Graphs, numbers, figuresWords, language, thoughts, ideas

What is Database?

An electronic place/system where data is stored for easy access, management, and updating.

What is DBMS?

A collection of interrelated data + programs to access that data. Primary goal: store and retrieve data conveniently and efficiently. Performs CRUD: Create, Read, Update, Delete.

DBMS vs File Systems

❓ Quick Q&A
Q: Why do we need DBMS over File Systems? +
To overcome 7 major disadvantages of file processing systems: data redundancy, difficulty in access, data isolation, integrity problems, atomicity issues, concurrent-access anomalies, and security problems.
Q: What is the difference between Data and Information? +
Data is raw, unorganized facts with no meaning. Information is processed, organized data that provides context and enables decision-making. Information depends on data, but not vice versa.

02 DBMS Architecture

▼

📌 Quick Summary

Three-level architecture (Physical, Logical, View) provides data abstraction. Includes schemas, data models, DB languages, DBA roles, and application architectures (1-tier, 2-tier, 3-tier).

Three Schema Architecture (View of Data)

Provides abstraction to simplify user interaction. Hides storage details.

📀 Physical / Internal Level
Lowest level — describes how data is actually stored (data structures, N-ary trees, compression, encryption). Physical schema defines physical storage.
🧠 Logical / Conceptual Level
Describes what data is stored and relationships. Used by DBA. Goal: ease of use.
👁️ View / External Level
Highest level — different views for different users. Hides irrelevant data. Provides security.

Instances & Schemas

Instance: Collection of info stored at a particular moment. Schema: Overall design (structural description) that doesn't change frequently.

3 types: Physical, Logical, View (subschemas). Physical data independence: changing physical schema should not affect logical schema.

Data Models

Collection of conceptual tools for describing data, relationships, semantics & constraints. E.g., ER model, Relational model, Object-oriented model.

Database Languages

📝 DDL (Data Definition Language)
Specifies database schema & consistency constraints.
🔍 DML (Data Manipulation Language)
Retrieval, insertion, deletion, updating of data. Query language is part of DML.

Database Administrator (DBA)

  • Schema Definition — designing the database structure
  • Storage structure & access methods
  • Authorization control — granting/revoking access
  • Routine maintenance — backups, security patches, upgrades

DBMS Application Architectures

🖥️ 1-Tier
Client, server & DB on same machine.
🔗 2-Tier
Client uses ODBC/JDBC to communicate with DB server.
🌐 3-Tier
Client → App Server → DB. Best for WWW. Advantages: Scalability, Data Integrity, Security.
❓ Quick Q&A
Q: What is physical data independence? +
The ability to change the physical schema (storage structures) without affecting the logical schema or application programs.
Q: What are the functions of a DBA? +
Schema definition, storage structure design, schema modifications, authorization control, and routine maintenance (backups, security patches, upgrades).

03 Entity-Relationship Model

▼

📌 Quick Summary

High-level data model based on real-world perception. Uses entities, attributes, relationships to design database blueprint via ER diagrams.

Entity

A "thing" or "object" distinguishable from others. Has physical existence, can be uniquely identified.

  • Strong Entity: Can be uniquely identified (has its own PK)
  • Weak Entity: Can't be uniquely identified; depends on a strong entity. E.g., Payment depends on Loan.

Attributes

Properties that describe an entity.

🔹 Simple
Can't be divided further. E.g., account number.
🔸 Composite
Can be divided into subparts. E.g., Name → first, middle, last.
🔹 Single-valued
Only one value. E.g., Student ID.
🔸 Multi-valued
Multiple values. E.g., phone numbers.
🔹 Derived
Derived from other attributes. E.g., Age from DOB.
🔸 NULL
"Not applicable" or "unknown" value.

Relationships

Association among entities.

  • Strong: Between independent entities
  • Weak: Between weak entity and its owner

Degree of Relationship

  • Unary: 1 entity participates. E.g., Employee manages Employee.
  • Binary: 2 entities. E.g., Student takes Course.
  • Ternary: 3 entities. E.g., Employee works-on Branch, works-on Job.

Mapping Cardinality

1:1 (One-to-One)
Citizen has Aadhar Card.
1:N (One-to-Many)
Citizen has Vehicle.
N:1 (Many-to-One)
Course taken by Professor.
M:N (Many-to-Many)
Student attends Course.

Participation Constraints

  • Total Participation: Every entity must be involved. E.g., Loan (can't exist without Customer).
  • Partial Participation: Not all entities are involved. E.g., Customer may or may not have a loan.
❓ Quick Q&A
Q: What's the difference between Strong and Weak Entity? +
Strong entity can be uniquely identified by its own attributes (has PK). Weak entity depends on a strong entity for identification — it doesn't have sufficient attributes for a unique key and uses a foreign key + partial discriminator.
Q: What is total vs partial participation? +
Total participation means every entity in the set must participate in at least one relationship (e.g., every loan must belong to a customer). Partial means entities may or may not participate (e.g., a customer may not have a loan).

04 Extended ER Features

▼

📌 Quick Summary

Extended ER features: Specialisation (top-down), Generalisation (bottom-up), Aggregation (treating relationships as entities), and Inheritance.

🔽 Specialisation
Top-down approach. Splitting entity set into sub-entity sets based on functionalities. E.g., Person → Customer, Student, Employee.
🔼 Generalisation
Bottom-up approach. Combining common attributes of entities into a superclass. E.g., Car, Jeep, Bus → Vehicle.
🔗 Aggregation
Treating relationships as higher-level entities (abstract entities) to show relationships among relationships.
🧬 Inheritance
Attribute & Participation Inheritance — child entity sets inherit attributes and relationship participation from parent.
❓ Quick Q&A
Q: What is the difference between Specialisation and Generalisation? +
Specialisation is a top-down approach where a high-level entity is split into sub-entities. Generalisation is bottom-up — combining common attributes of multiple entities into a superclass. Both use "is-a" relationships and attribute inheritance.

05 Relational Model

▼

📌 Quick Summary

Organizes data in relations (tables). Core concepts: tuple (row), attribute (column), domain, keys (Super, Candidate, Primary, Foreign), integrity constraints.

Key Terms

📋 Relation
Table with unique name.
📌 Tuple
A single row / record.
📊 Attribute
Column of a table.
📐 Domain
Set of permitted values for an attribute.
📏 Degree
Number of attributes/columns.
🔢 Cardinality
Number of tuples/rows.

Properties of a Relation

  • Each relation name is unique
  • Values must be atomic (can't be broken further)
  • Each attribute name must be unique
  • Each tuple must be unique
  • Row/Column sequence has no significance
  • Must follow integrity constraints

Keys in Relational Model

KeyDescription
🔑 Super KeyAny set of attributes that can uniquely identify each tuple.
🏆 Candidate KeyMinimal subset of super key (no redundant attributes). Cannot be NULL.
⭐ Primary Key (PK)Selected from Candidate Keys. Has the least attributes.
🔄 Alternate KeyAll Candidate Keys except PK.
🔗 Foreign Key (FK)Creates relation between tables. References PK of another table.
🧩 Composite KeyPK formed using ≥2 attributes.
🤖 Surrogate KeyAuto-generated synthetic PK (usually integer).

Integrity Constraints

  • Domain Constraints: Restrict data types & value ranges
  • Entity Constraints: PK ≠ NULL
  • Referential Constraints: FK must match PK in parent table or be NULL
  • Key Constraints: NOT NULL, UNIQUE, DEFAULT, CHECK, PRIMARY KEY, FOREIGN KEY
❓ Quick Q&A
Q: What is the difference between Super Key and Candidate Key? +
Super Key is any set of attributes that uniquely identifies tuples. Candidate Key is a minimal Super Key — it has no redundant attributes. Every Candidate Key is a Super Key, but not every Super Key is a Candidate Key.
Q: What is Referential Integrity? +
Ensures that a Foreign Key value in one table must either be NULL or match a Primary Key value in the referenced table. This maintains consistency between related tables.

06 ER to Relational Transformation

▼

📌 Quick Summary

Converting ER diagram to relational tables. Each entity, attribute, and relationship maps to specific table structures.

ER ComponentRelational Mapping

Weak EntityTable with all attributes + FK to Strong Entity. PK = {FK + Partial discriminator}.
Simple AttributeDirect column in the table.
Composite AttributeSeparate columns for each subpart. E.g., Address → address_street, address_city.
Multi-valued AttributeNew table with {FK + attribute_value} as PK.

Q: How do you map a multi-valued attribute to a relation? +
Create a new table named after the attribute. Include the PK of the original entity as a FK, and the multi-valued attribute itself as a column. The PK of the new table is {FK + attribute_column}.

A view is a virtual table created by a query. Doesn't store data itself — reflects changes from underlying tables.

❓ Quick Q&A
Q: What's the difference between WHERE and HAVING? +
WHERE filters rows before GROUP BY, while HAVING filters groups after GROUP BY. WHERE can be used with SELECT, UPDATE, DELETE. HAVING requires GROUP BY.
Q: What is the difference between DELETE and TRUNCATE? +
DELETE is a DML command that can use WHERE to delete specific rows (can be rolled back). TRUNCATE is a DDL command that removes all rows and cannot be rolled back. TRUNCATE is faster.
Q: What is the difference between SQL and MySQL? +
SQL is Structured Query Language used to perform CRUD operations. MySQL is an RDBMS (Relational Database Management System) that uses SQL to manage data.

08 Normalisation

▼

📌 Quick Summary

Normalisation minimizes data redundancy and eliminates anomalies (Insertion, Deletion, Updation) by dividing large tables into smaller, related tables.

Functional Dependency (FD)

X → Y means Y is functionally dependent on X. X = Determinant, Y = Dependent.

Types of FD

  • Trivial FD: B is subset of A. E.g., A→A, B→B
  • Non-trivial FD: B is NOT a subset of A. E.g., A→B where A ∩ B = ∅

Armstrong's Axioms

🔄 Reflexive
If A ⊇ B then A → B.
📈 Augmentation
If A→B then AX→BX.
🔗 Transitivity
If A→B and B→C then A→C.

Anomalies (Problems with Redundancy)

➕ Insertion Anomaly
Cannot insert data without presence of other data.
➖ Deletion Anomaly
Deleting data causes unintended loss of other data.
✏️ Updation Anomaly
Updating requires multiple rows to be updated → data inconsistency risk.

Normal Forms

Normal FormCondition
1NFAll attribute values must be atomic (no multi-valued attributes).
2NF1NF + No partial dependency — all non-prime attributes fully dependent on PK.
3NF2NF + No transitive dependency — non-prime attributes must not determine other non-prime attributes.
BCNF3NF + For every FD X→Y, X must be a super key.
❓ Quick Q&A
Q: What is partial dependency in 2NF? +
When a non-prime attribute depends on only part of a composite primary key (not the full key). To satisfy 2NF, all non-prime attributes must be fully functionally dependent on the entire PK.
Q: What is the difference between 3NF and BCNF? +
In 3NF, transitive dependencies are not allowed (non-prime → non-prime). In BCNF, for every FD X→Y, X must be a super key. BCNF is stricter — it also prevents prime attribute determining another prime attribute if the determinant isn't a super key.

09 Transactions

▼

📌 Quick Summary

A transaction is a logical unit of work with one or more SQL statements. ACID properties ensure data integrity.

ACID Properties

⚛️ Atomicity
All operations succeed or none are reflected (all-or-nothing).
✅ Consistency
DB must be consistent before and after transaction. Integrity must be maintained.
🔒 Isolation
Concurrent transactions don't interfere with each other. Each appears to execute alone.
💾 Durability
Once committed, changes persist even after system failures.

Transaction States

🟢 Active
Read/write operations executing.
🟡 Partially Committed
Executed, changes in buffer (main memory).
🟢 Committed
Changes permanent on DB. Can't rollback.
🔴 Failed
Failure occurred during execution.
⚪ Aborted
Changes rolled back. DB restored to prior state.
🔚 Terminated
Committed or Aborted.

Flow: Active → Partially Committed → Committed (success) OR Active → Failed → Aborted → Terminated

❓ Quick Q&A
Q: What happens if a transaction fails after partial execution? +
The transaction moves to Failed state, then Aborted state where all changes are rolled back using undo logs. The DB returns to its state before the transaction began.
Q: What does Isolation mean in ACID? +
Multiple transactions executing concurrently should not interfere with each other. Each transaction appears to execute in isolation, unaware of other concurrent transactions.

10 Atomicity & Durability Implementation

▼

📌 Quick Summary

Recovery mechanisms implement Atomicity and Durability. Two main approaches: Shadow-copy scheme and Log-based recovery.

1. Shadow-Copy Scheme

  • Creates a complete copy of DB for each transaction
  • db-pointer points to current DB copy on disk
  • Transaction updates new copy; old copy (shadow) untouched
  • On abort: Delete new copy
  • On commit: Write pages to disk → Update db-pointer → Delete old copy
  • ⚠️ Inefficient — copies entire DB for every transaction

2. Log-Based Recovery

Maintains a log of all DB modifications in stable storage before applying actual changes.

📝 Deferred Modifications
All writes deferred until transaction completes. On crash — ignore logs. On complete — redo.
📝 Immediate Modifications
Writes happen while transaction is active (uncommitted). On failure — undo using old values. On complete — redo using new values.
❓ Quick Q&A
Q: What is the purpose of a log in DB recovery? +
The log records all DB modifications in stable storage before they are applied. It ensures atomicity (by allowing undo of failed transactions) and durability (by allowing redo of committed transactions after system crashes).

11 Indexing in DBMS

▼

📌 Quick Summary

Indexing optimizes query performance by minimizing disk accesses. Like a book's index — provides fast data lookup.

Key Concepts

  • Search Key: Copy of PK or candidate key
  • Data Reference: Pointer to disk block address
  • Index file is always sorted
  • Indexing is optional — secondary access method

Dense vs Sparse Index

🔵 Dense Index
Index record for every search key value. More space, faster lookup.
🟢 Sparse Index
Index record for only some search key values. Less space, but slower.

Primary Index (Clustering Index)

  • Data file sorted sequentially on search key
  • Based on Key attribute: Sparse index, entries = number of blocks
  • Based on Non-key attribute: Dense index, entries = unique values

Secondary Index (Non-Clustering)

  • Data file is unsorted
  • Always dense index — entries = number of records
  • One primary index exists, others are secondary

Multi-level Index

When single-level index is too large for efficient binary search, break it into multiple levels (like a tree).

⚠️ Trade-off: Indexing speeds up SELECT/READ operations but slows down INSERT, DELETE, UPDATE. Also requires additional storage space.
❓ Quick Q&A
Q: What is the difference between Primary and Secondary Index? +
Primary index is on a sorted data file — the search key defines the sequential order. Secondary index is on an unsorted data file. Primary can be sparse or dense; secondary is always dense.

13 Types of Databases

▼

📌 Quick Summary

Databases categorized by their data models: Relational, Object-Oriented, NoSQL, Hierarchical, Network.

14 Clustering in DBMS

▼

📌 Quick Summary

Combining multiple servers to manage a single database. Replicates same dataset across servers for redundancy, load balancing & high availability.

Advantages

💾 Data Redundancy
Same data on multiple servers. Required redundancy (not anomaly). If one server fails, data is available elsewhere.
⚖️ Load Balancing
Workload distributed across cluster servers. More users supported, handles traffic spikes.
🟢 High Availability
Database remains accessible even if some servers fail. Minimal to no system downtime.

How It Works

All requests split across multiple computers. If one node fails, request is handled by another node. Combines load balancing + failover.

15 Partitioning & Sharding

▼

📌 Quick Summary

Dividing large databases into smaller, manageable slices. Vertical = column-wise, Horizontal = row-wise. Sharding implements horizontal partitioning.

Types of Partitioning

↕️ Vertical Partitioning
Splitting relation column-wise. May need to access different servers for a complete tuple.
↔️ Horizontal Partitioning
Splitting relation row-wise. Independent chunks stored on different servers.

When to Partition?

  • Dataset is too large to manage efficiently
  • Single DB server response time is high

Advantages

  • ✅ Parallelism — queries run faster
  • ✅ Availability — failure in one partition doesn't affect others
  • ✅ Performance — smaller datasets are faster
  • ✅ Manageability — easier to handle smaller pieces
  • ✅ Cost — cheaper than vertical scaling

Sharding

Technique to implement Horizontal Partitioning. Data split across multiple DB instances with a routing layer directing requests.

✅ Pros
Scalability, Availability
⚠️ Cons
Complexity (partition mapping, routing), Re-sharding needs, Not suited for analytical queries (Scatter-Gather problem)
❓ Quick Q&A
Q: What is the difference between Partitioning and Sharding? +
Partitioning is the general concept of dividing a database into smaller pieces. Sharding is a specific technique that implements horizontal partitioning across multiple database instances/servers, with a routing layer to direct requests to the correct shard.

16 CAP Theorem

▼

📌 Quick Summary

In a distributed system, you can only have 2 of 3 properties: Consistency, Availability, Partition Tolerance. Fundamental trade-off in distributed databases.

The Three Properties

✅ Consistency (C)
All nodes see the same data at the same time. Every read returns the most recent write.
🟢 Availability (A)
System remains operational. Every request gets a response (even if not the latest data).
🔗 Partition Tolerance (P)
System continues to work despite communication breaks between nodes. Must replicate across nodes.

CAP Combinations

TypePropertiesDescriptionExamples
CAConsistency + AvailabilityNo partition tolerance. Not practical for distributed systems since partitions are inevitable.MySQL, PostgreSQL (with replication)
CPConsistency + Partition ToleranceAvailability sacrificed during partitions. Inconsistent nodes turned off until resolved. Good for banking.MongoDB
APAvailability + Partition ToleranceConsistency sacrificed. All nodes stay available but may have stale data. Eventual consistency. Good for social media.Cassandra, DynamoDB
❓ Quick Q&A
Q: Why can't we have all three (C, A, P) in a distributed system? +
During a network partition, you must choose: either keep all nodes available (but they might return stale data → no consistency) or ensure all nodes have consistent data (but some nodes may be unavailable). You can't have both simultaneously during a partition.
Q: Which CAP combination is best for a banking system vs social media? +
Banking → CP (Consistency is critical, availability can temporarily suffer). Social media → AP (Availability is more important, eventual consistency is acceptable).

17 Master-Slave Database Concept

▼

📌 Quick Summary

Optimization pattern where Master handles writes (true/latest data) and Slaves handle reads. Replication syncs data from Master to Slaves.

How It Works

👑 Master DB
Handles all write operations (INSERT, UPDATE, DELETE). Contains the true/latest data.
🛡️ Slave DB(s)
Handle read operations (SELECT). Data replicated from Master. Multiple slaves for load distribution.

Benefits

  • ✅ Load Distribution: Reads distributed across slaves, reducing master overload
  • ✅ High Availability: If master fails, a slave can be promoted
  • ✅ Reduced Latency: Better response times for read-heavy applications
  • ✅ Scalability: Add more slaves as read demand grows
💡 Note: Replication can be synchronous (real-time) or asynchronous (delayed). This is also called CQRS (Command Query Responsibility Segregation) pattern.

⭐ DB Scaling Patterns (Step-by-Step)

▼

📌 Quick Summary

Progressive scaling strategy as an app grows — from query optimization to multi-data-center deployment.

  1. Pattern 1 — Query Optimization & Connection Pool: Cache frequent non-dynamic data, introduce DB redundancy, use connection pool libraries.
  2. Pattern 2 — Vertical Scaling (Scale-up): Upgrade machine (RAM, SSD). Cost increases exponentially beyond a point.
  3. Pattern 3 — CQRS: Separate read/write operations. Primary for writes, replicas for reads.
  4. Pattern 4 — Multi-Primary Replication: All machines can act as primary & replica. Write to any node, read from fastest responder.
  5. Pattern 5 — Partitioning by Functionality: Separate DB schemas for different functionalities (location, users, etc.). App layer joins results.
  6. Pattern 6 — Sharding (Horizontal Scaling): Multiple machines, same schema, each holds part of data. Requires routing layer.
  7. Pattern 7 — Data Centre Wise Partition: Distribute traffic across continents. Cross-data-centre replication for disaster recovery.

💡 Key Insight: Start simple (Pattern 1) and progressively apply more complex patterns as needed. "No Pain, No Gain" — especially with sharding.

❓ Common Interview Q&A

▼
Q: What is a Database Transaction? +
A transaction is a logical unit of work containing one or more SQL statements. All statements either complete successfully (committed) or none are applied (rolled back). Transactions follow ACID properties.
Q: Explain the difference between DROP, TRUNCATE and DELETE. +
DROP: DDL — removes entire table structure and data. Can't rollback. TRUNCATE: DDL — removes all rows, resets table but keeps structure. Faster, minimal logging. DELETE: DML — removes rows (can use WHERE). Can be rolled back. Triggers fire.
Q: What is the difference between UNION and UNION ALL? +
UNION returns only distinct rows from combined result sets. UNION ALL returns all rows including duplicates. UNION ALL is faster because it doesn't perform deduplication.
Q: What is a correlated subquery? +
A correlated subquery executes once for each row of the outer query. Unlike normal subqueries (which run once), the inner query depends on values from the outer query. This makes it slower but more flexible for certain problems.
Q: What is the difference between OLTP and OLAP? +
OLTP (Online Transaction Processing): Handles day-to-day transactions (INSERT, UPDATE, DELETE). Many small transactions, real-time. OLAP (Online Analytical Processing): Handles complex analytical queries. Fewer but larger queries, aggregated data, historical analysis. E.g., OLTP = banking transactions, OLAP = annual sales report.
Q: What is a Deadlock in DBMS? +
A deadlock occurs when two or more transactions are waiting for each other to release locks, creating a cycle of dependencies. E.g., T1 holds Lock A and waits for Lock B; T2 holds Lock B and waits for Lock A. Resolved by deadlock detection & rollback of one transaction.
Q: What are ACID properties? Why are they important? +
ACID ensures reliable processing of database transactions: Atomicity (all-or-nothing), Consistency (integrity maintained), Isolation (concurrent transactions don't interfere), Durability (committed changes persist). They prevent data corruption and ensure reliability.
Q: What are the advantages of Normalization? +
Normalization minimizes data redundancy, eliminates anomalies (insertion, deletion, updation), ensures data consistency, and maintains greater overall database organization. It divides larger tables into smaller, related tables linked by relationships.
Q: What is the difference between CHAR and VARCHAR? +
CHAR(n): Fixed-length string. Always uses n characters (padded with spaces). Faster for fixed-length data. VARCHAR(n): Variable-length string. Uses only as many characters as needed (up to n). Saves space. Prefer VARCHAR for variable-length data.
Q: What is data independence? +
Data independence is the ability to change the schema at one level without affecting the schema at the next higher level. Physical data independence: Change physical storage without affecting logical schema. Logical data independence: Change logical schema without affecting external views.

📐 Entity-Relationship Diagrams

Q1: What is an Entity-Relationship (ER) diagram, and what is its purpose? +
An ER diagram is a graphical representation of entities and their relationships in a database. Its purpose is to visually design the database schema, define entities, attributes, and relationships before implementation.
Q2: Describe the main components of an ER diagram. +
Main components: Entities (rectangles), Attributes (ellipses), Relationships (diamonds), Cardinality constraints (1:1, 1:N, M:N), and Connecting lines.
Q3: What is an entity, and how is it represented in an ER diagram? +
An entity is a real-world object or concept distinguishable from others. Represented as a rectangle in ER diagrams. Example: Student, Course, Employee.
Q4: Explain the different types of entities commonly used in ER diagrams. +
Strong entities (can exist independently, have their own PK) and Weak entities (depend on strong entity for identification, use partial discriminator).
Q5: What is an attribute, and how is it represented in an ER diagram? +
An attribute describes properties of an entity. Represented as ellipses connected to entities. Key attributes are underlined. Multi-valued attributes use double ellipses.
Q6: Describe the different types of attributes, including simple, composite, and derived. +
Simple: indivisible (e.g., age). Composite: divisible (e.g., name splits into first/last). Derived: calculated from others (e.g., age from DOB). Single-valued vs multi-valued. NULL vs NOT NULL.
Q7: What is a relationship, and how is it represented in an ER diagram? +
A relationship is an association between entities. Represented as a diamond shape connected to participating entities by lines.
Q8: Explain the cardinality and participation constraints in relationships. +
Cardinality: max number of relationships an entity can participate in (1:1, 1:N, N:M). Participation: total (every entity must participate) vs partial (entities may or may not).
Q9: Describe the different types of relationships (one-to-one, one-to-many, many-to-many). +
1:1 — one entity relates to one other (e.g., Person has Passport). 1:N — one relates to many (e.g., Department has Employees). M:N — many relate to many (e.g., Students enroll in Courses).
Q10: What are weak entities, and how are they represented in an ER diagram? +
Weak entities cannot be uniquely identified by their own attributes. They depend on a strong entity (owner). Represented by a double rectangle. Their discriminator (partial key) is underlined with a dashed line.
Q11: Explain the concept of identifying and non-identifying relationships. +
Identifying relationship: connects weak entity to its strong owner (child cannot exist without parent). Non-identifying: both entities can exist independently; FK can be NULL.
Q12: What is an associative entity, and when is it used? +
An associative entity (gerund) resolves M:N relationships by introducing an intermediate entity. It inherits PKs from both participating entities. Used when the relationship itself has attributes.
Q13: How do you denote specialization and generalization in an ER diagram? +
Using a triangle (ISA hierarchy). Specialization: top-down splitting (Person → Student, Employee). Generalization: bottom-up combining (Car, Bus → Vehicle).
Q14: What are the key differences between an ER diagram and a relational schema? +
ER diagram is conceptual/high-level, shows entities & relationships. Relational schema is logical, shows tables with columns, PKs, and FKs. ER maps to tables in relational model.
Q15: Describe the process of converting an ER diagram into a relational schema. +
Each entity → table. Attributes → columns. PK → table's PK. Relationships → FK references. Weak entities → own table with FK + partial key. M:N → junction table. Multi-valued → separate table.
Q16: How do you validate the correctness of an ER diagram? +
Check: all entities have PKs, relationships have correct cardinality, weak entities have identifying relationships, no redundant relationships, all required attributes captured, normalization rules followed.
Q17: What are some best practices for designing effective ER diagrams? +
Use meaningful naming, identify all entities first, define relationships clearly, minimize redundancy, use appropriate cardinality, document constraints, review with stakeholders, keep diagrams manageable.
Q18: Explain the concept of aggregation in ER diagrams. +
Aggregation treats a relationship as an abstract higher-level entity. Used when you need to relate a relationship to another entity. Represented by a dashed rectangle around the relationship.
Q19: How can you represent constraints and business rules in an ER diagram? +
Through cardinality constraints (1:1, 1:N, M:N), participation constraints (total/partial), key constraints (underlined attributes), and domain constraints on attribute values.
Q20: What are some common tools used for creating and manipulating ER diagrams? +
MySQL Workbench, draw.io, Lucidchart, ER/Studio, dbdiagram.io, Oracle SQL Developer Data Modeler, Microsoft Visio, pgAdmin.

🗄️ DBMS Fundamentals & Types

Q21: What is a Database Management System (DBMS), and what are its main functions? +
DBMS is software that manages, stores, retrieves, and manipulates data. Main functions: data storage/retrieval, transaction management, concurrency control, security, integrity enforcement, backup/recovery.
Q22: Describe the key components of a DBMS architecture. +
Components: Storage engine, Query processor, Transaction manager, Lock manager, Recovery manager, Buffer manager, Data dictionary, Security manager.
Q23: What are the advantages of using a DBMS over traditional file-based systems? +
Reduced redundancy, data consistency, data independence, concurrent access, security, integrity constraints, backup/recovery, easy querying, multiple user access.
Q24: Explain the concept of data independence in the context of DBMS. +
Data independence allows changing the schema at one level without affecting higher levels. Physical independence: change storage without affecting logical. Logical independence: change logical schema without affecting views.
Q25: Describe the role of the Data Definition Language (DDL) in a DBMS. +
DDL defines the database schema: CREATE, ALTER, DROP tables/indexes/views. Also specifies constraints (PK, FK, UNIQUE, CHECK). DDL changes are auto-committed.
Q26: What is a data model, and how does it relate to a DBMS? +
A data model is a collection of concepts for describing data, relationships, semantics, and constraints. Examples: Relational, ER, Object-oriented, Hierarchical. DBMS implements a specific data model.
Q27: Discuss the difference between a relational database and a NoSQL database. +
Relational: fixed schema, tables, ACID, SQL, vertical scaling. NoSQL: flexible schema, various models (document, key-value, graph), BASE, horizontal scaling, suited for big data.
Q28: How does a DBMS ensure data integrity and security? +
Integrity: constraints (PK, FK, CHECK, UNIQUE, NOT NULL), triggers, transactions. Security: authentication, authorization (GRANT/REVOKE), encryption, auditing, views for data hiding.
Q29: Explain the concept of transaction management in a DBMS. +
A transaction is a logical unit of work. Transaction management ensures ACID properties: Atomicity (all-or-nothing), Consistency (valid state), Isolation (concurrent execution safe), Durability (committed changes persist).
Q30: What are the common types of users in a DBMS environment? +
Database Administrator (DBA), Application Programmers, End Users (naive/sophisticated), Database Designers, System Analysts.
Q31: What is a relational database management system (RDBMS), and how does it differ from other types? +
RDBMS stores data in tables with rows/columns, enforces relationships via FKs, uses SQL. Differs from hierarchical (tree), network (graph), object-oriented (objects) by using set theory and relational algebra.
Q32: Describe the characteristics and advantages of a hierarchical database system. +
Tree structure, parent-child relationships, one-to-many only, fast access via paths. Advantages: simple, fast for predefined queries. Disadvantage: inflexible, no many-to-many without redundancy.
Q33: Explain the concept of network database management systems (NDBMS). +
Extension of hierarchical model allowing many-to-many via sets. Uses CODASYL standard. Child can have multiple parents. More flexible but complex to design and maintain.
Q34: What is an object-oriented database management system (OODBMS), and when is it used? +
OODBMS stores data as objects (like OOP). Supports inheritance, encapsulation, polymorphism. Used for complex data: CAD/CAM, multimedia, scientific applications. Example: ObjectDB.
Q35: Discuss the characteristics of a document-oriented database management system. +
Stores semi-structured data as documents (JSON, BSON, XML). Schema-flexible. Supports nested data, embedded documents. Examples: MongoDB, CouchDB. Good for content management, catalogs.
Q36: Explain the concept of a columnar database management system (CDBMS). +
Stores data by columns instead of rows. Efficient for analytical queries/aggregations only on specific columns. Better compression. Examples: Apache Cassandra, Redshift, Bigtable.
Q37: What is a graph database management system (GDBMS), and how does it represent data? +
GDBMS uses nodes (entities) and edges (relationships). Optimized for connected data and relationship traversal. Examples: Neo4j, Amazon Neptune. Used for social networks, fraud detection, recommendations.
Q38: Describe the characteristics of an in-memory database management system. +
Stores data primarily in RAM for ultra-fast access. Microsecond latency. Used for real-time analytics, caching, session management. Examples: Redis, SAP HANA. Trade-off: volatility (requires persistence strategy).
Q39: What is a time-series database management system, and what types of data does it handle? +
Optimized for time-stamped data (metrics, sensor readings, logs). Features: automatic retention, downsampling, time-based queries. Examples: InfluxDB, TimescaleDB, Prometheus.
Q40: Discuss the role of distributed database management systems (DDBMS) in modern computing. +
DDBMS manages data across multiple physically separate sites. Advantages: reliability, availability, scalability, local autonomy. Challenges: distributed transactions, concurrency, consistency (CAP theorem).

🔑 Keys & Constraints

Q41: What is a primary key, and why is it important in a relational database? +
A primary key uniquely identifies each row in a table. It enforces entity integrity, ensures no duplicate rows, and provides a reliable way to reference specific records.
Q42: Describe the characteristics of a primary key. +
Unique (no duplicate values), NOT NULL, minimal (use fewest attributes possible), stable (values rarely change), single per table.
Q43: How is a primary key different from other types of keys in a database? +
PK is the chosen candidate key. Super Key uniquely identifies rows (can have extra attributes). Candidate Key is a minimal Super Key. FK references PK of another table. Alternate Key = candidate keys not chosen as PK.
Q44: Can a primary key contain null values? Why or why not? +
No. Primary keys cannot be NULL because NULL represents unknown/missing value, which cannot guarantee uniqueness. Entity integrity requires every row to be uniquely identifiable.
Q45: Explain the concept of surrogate keys and their role in primary key design. +
Surrogate key is an artificial, system-generated unique identifier (e.g., auto-increment INT, UUID). Not derived from business data. Advantages: stable, simple, unaffected by business changes.
Q46: What are the advantages of using a natural key as opposed to a surrogate key? +
Natural key is meaningful business data (e.g., SSN, email). Advantages: no extra column needed, already exists, meaningful for queries. Disadvantage: can change, may be large.
Q47: Discuss the process of selecting a suitable primary key for a table. +
Evaluate candidate keys: uniqueness, stability, simplicity, size. Prefer single-column keys. Surrogate keys often preferred for stability. Ensure key satisfies future growth requirements.
Q48: How does a primary key enforce entity integrity in a relational database? +
Entity integrity rule: no PK attribute can be NULL, and all PK values must be unique. DBMS automatically enforces these constraints, preventing duplicate or incomplete rows.
Q49: What happens if you try to insert a duplicate value into a primary key column? +
The DBMS rejects the insert with a constraint violation error (e.g., 'Duplicate entry for PRIMARY KEY'). The insert fails and the row is not added.
Q50: Can a table have multiple primary keys? Explain. +
No, a table can have only one primary key. However, a primary key can be composite (multiple columns). The multiple candidate keys not chosen as PK are called alternate keys.
Q51: What is a foreign key, and how is it used in a relational database? +
A foreign key is a column(s) referencing the PK of another table. It establishes relationships between tables and enforces referential integrity.
Q52: Describe the relationship between a foreign key and a primary key. +
FK in child table references PK in parent table. This creates a parent-child relationship ensuring that values in FK column must either match existing PK values or be NULL.
Q53: How does a foreign key enforce referential integrity between related tables? +
Referential integrity rule: FK value must exist in referenced PK or be NULL. Prevents orphan records. Actions: CASCADE, SET NULL, SET DEFAULT, NO ACTION on UPDATE/DELETE.
Q54: Can a foreign key contain null values? When is this allowed? +
Yes, FK can contain NULL unless NOT NULL is specified. This represents an optional relationship where a row is not related to any parent row.
Q55: Explain the difference between a foreign key constraint and a foreign key index. +
Constraint enforces referential integrity (rules). Index improves lookup performance on FK columns. The constraint is required for referential integrity; the index is optional but recommended.
Q56: Discuss the process of creating a foreign key constraint in a table. +
Use ALTER TABLE or CREATE TABLE with FOREIGN KEY clause: FOREIGN KEY (col) REFERENCES parent_table(pk_col) [ON DELETE/UPDATE CASCADE/SET NULL/SET DEFAULT].
Q57: What happens if you try to insert a value into a foreign key column that doesn't exist in the referenced table? +
The DBMS rejects the insert with a foreign key constraint violation error. The FK value must match an existing PK in the referenced table (or be NULL if allowed).
Q58: Can a table have multiple foreign keys? Are there any limitations? +
Yes, a table can have multiple FKs referencing different tables. Limitations: each FK must reference a PK/UNIQUE in its parent table, all referential rules apply independently.
Q59: How do you handle cascading updates and deletes with foreign key constraints? +
Using ON UPDATE CASCADE (updates FK when PK changes) and ON DELETE CASCADE (deletes child rows when parent deleted). Alternative: ON DELETE SET NULL, SET DEFAULT, or NO ACTION.
Q60: What are the benefits of using foreign keys in database design? +
Enforces referential integrity, prevents orphan records, maintains data consistency, documents relationships between tables, enables cascading operations, supports normalization.

📏 Normalization

Q61: What is normalization and why is it important in database design? +
Normalization systematically eliminates data redundancy and anomalies. It divides large tables into smaller, related tables. Benefits: reduces storage, eliminates update/insert/delete anomalies, improves data integrity.
Q62: Explain the purpose of normal forms in the context of database normalization. +
Normal forms are progressive levels of data organization. Each normal form adds constraints to reduce redundancy. Higher NF = less redundancy but more tables. Trade-off between normalization and performance.
Q63: Can you describe the process of normalization and its stages? +
Process: analyze FDs → check 1NF (atomic values) → 2NF (no partial dependency) → 3NF (no transitive dependency) → BCNF (every determinant is a super key). Higher forms exist (4NF, 5NF) but rarely used.
Q64: What are the different normal forms, and what are their characteristics? +
1NF: atomic columns. 2NF: 1NF + full FD on PK. 3NF: 2NF + no transitive FD. BCNF: 3NF + every determinant is super key. 4NF: BCNF + no multi-valued dependencies. 5NF: join dependencies.
Q65: How does normalization help in reducing redundancy and improving data integrity? +
By splitting tables to ensure each fact is stored once. Reduces update anomalies (changing one value in one place). Prevents insertion anomaly (can't add data without other data). Prevents deletion anomaly.
Q66: Discuss the advantages and disadvantages of normalization. +
Advantages: reduced redundancy, improved integrity, easier maintenance, flexible schema. Disadvantages: more joins (slower queries), complex design, may require denormalization for performance.
Q67: Explain the concept of functional dependencies and how they relate to normalization. +
FD X→Y means X uniquely determines Y. Core to normalization — normal forms are defined based on FDs. Used to identify partial and transitive dependencies to eliminate.
Q68: What is the difference between partial dependency and transitive dependency? +
Partial dependency: non-key attribute depends on only part of composite PK (violates 2NF). Transitive dependency: non-key attribute depends on another non-key attribute (violates 3NF).
Q69: Can you provide examples of violations of different normal forms? +
Unnormalized: multiple phone numbers in one column (violates 1NF). Partial: student_name depends only on student_id, not on composite PK (violates 2NF). Transitive: city depends on zip_code, not PK (violates 3NF).
Q70: How do you determine which normal form a database table is currently in? +
Check: atomic values → 1NF. Full FD on PK → 2NF. No transitive FD → 3NF. All determinants are super keys → BCNF. Identify all FDs and check against NF definitions.
Q71: In what scenarios would denormalization be appropriate? +
When query performance is critical and joins are too slow. Read-heavy applications, reporting/analytics, data warehousing. Trade-offs: increased redundancy, update complexity.
Q72: Discuss the role of candidate keys and primary keys in normalization. +
Candidate keys determine all other attributes (they are determinants). PK is chosen candidate key. Normal forms check dependencies on candidate keys and their subsets.
Q73: How does normalization affect query performance and database maintenance? +
Normalization increases joins (slower reads) but reduces update anomalies (faster writes). Maintenance easier due to less redundancy. Trade-off often resolved via strategic denormalization.
Q74: Explain the process of database normalization in a real-world example. +
Example: Order table with (order_id, customer_name, product_name, product_price, order_date). Split into: Orders(order_id, customer_id, date), Customers(customer_id, name), Order_Items(order_id, product_id), Products(product_id, name, price).
Q75: How does normalization contribute to better scalability and flexibility? +
Normalized schema adapts better to change (add attributes in one place). Easier to scale because data is logically separated. Less data duplication means less storage and easier distribution.

💻 SQL

Q76: What is SQL and what does it stand for? +
SQL stands for Structured Query Language. It's a standard language for managing relational databases: querying, inserting, updating, and deleting data.
Q77: Name different categories of SQL commands. +
DDL (CREATE, ALTER, DROP), DML (SELECT, INSERT, UPDATE, DELETE), DCL (GRANT, REVOKE), TCL (COMMIT, ROLLBACK, SAVEPOINT).
Q78: What is a database table? +
A table is a collection of related data organized in rows and columns. Each column represents an attribute, each row represents a record. The fundamental storage unit in relational databases.
Q79: What is a DBMS and how does it differ from an RDBMS? +
DBMS manages databases (any model). RDBMS is a specific type of DBMS using the relational model (tables, relationships, SQL). All RDBMS are DBMS but not vice versa. RDBMS enforces relationships via FKs.
Q80: What are the different types of keys in SQL? +
Primary Key, Foreign Key, Unique Key, Composite Key, Candidate Key, Alternate Key, Super Key, Surrogate Key.
Q81: What is a view? +
A view is a virtual table based on a SELECT query. Doesn't store data physically. Provides security (restrict column access), simplifies complex queries, and presents different perspectives of data.
Q82: What are constraints in SQL? +
Rules enforced on data columns. Types: NOT NULL, UNIQUE, PRIMARY KEY, FOREIGN KEY, CHECK, DEFAULT. Ensure data integrity and consistency.
Q83: What is the difference between CHAR and VARCHAR data types? +
CHAR has fixed length (padded with spaces). VARCHAR has variable length (only uses needed space). CHAR is faster for fixed-length data. VARCHAR is storage-efficient for variable-length data.
Q84: Explain the difference between DROP, TRUNCATE, and DELETE. +
DROP: removes entire table structure + data (DDL). TRUNCATE: removes all rows, keeps structure (DDL, faster, can't rollback). DELETE: removes rows selectively (DML, can rollback, fires triggers).
Q85: What is a JOIN in SQL? +
JOIN combines rows from two or more tables based on a related column. Types: INNER, LEFT, RIGHT, FULL, CROSS, SELF JOIN. Enables querying data across normalized tables.
Q86: What is the difference between INNER JOIN and OUTER JOIN? +
INNER JOIN returns only matching rows from both tables. OUTER JOIN returns matching rows plus non-matching rows from one/both tables: LEFT (all left), RIGHT (all right), FULL (all from both).
Q87: What is a self-join? +
A self-join joins a table with itself using aliases. Used when a table has a foreign key referencing its own primary key (e.g., employees with manager_id referencing employee_id).
Q88: What is the difference between UNION and UNION ALL? +
UNION removes duplicates (slower). UNION ALL keeps all rows (faster). Both combine result sets vertically — requires same number/type of columns.
Q89: What is a subquery? +
A subquery is a query nested inside another query. Can be used in SELECT, FROM, WHERE, HAVING clauses. Executes before the outer query (non-correlated) or with each row (correlated).
Q90: What is a correlated subquery? +
A correlated subquery references columns from the outer query and executes once for each outer row. Example: SELECT * FROM emp e WHERE salary > (SELECT AVG(salary) FROM emp WHERE dept_id = e.dept_id).
Q91: What are aggregate functions in SQL? +
Functions that perform calculations on multiple rows and return a single value. Common: COUNT, SUM, AVG, MIN, MAX. Often used with GROUP BY. NULLs are ignored except COUNT(*).
Q92: What is the difference between WHERE and HAVING clauses? +
WHERE filters rows before GROUP BY. HAVING filters groups after GROUP BY. WHERE can't use aggregate functions. HAVING can. WHERE is used with SELECT, UPDATE, DELETE. HAVING only with SELECT.
Q93: What are GROUP BY and ORDER BY clauses? +
GROUP BY groups rows with same values and is used with aggregates. ORDER BY sorts result set (ASC/DESC). GROUP BY happens before ORDER BY in query execution.
Q94: What is the purpose of the DISTINCT keyword? +
DISTINCT removes duplicate rows from the result set. SELECT DISTINCT returns only unique combinations of selected columns.
Q95: What are wildcards in SQL? +
Wildcards used with LIKE operator: % (matches any sequence of chars), _ (matches single char). Example: WHERE name LIKE 'A%' finds names starting with A.
Q96: What is an index in SQL? +
An index is a database structure that speeds up data retrieval. Works like a book index. Types: clustered (determines physical order, one per table) and non-clustered (separate structure).
Q97: What is a stored procedure? +
A stored procedure is a pre-compiled SQL code block stored in the database. Accepts parameters, returns results. Benefits: performance (compiled), reusability, security, reduced network traffic.
Q98: What is a trigger in SQL? +
A trigger is an automatic SQL code execution in response to events (INSERT, UPDATE, DELETE) on a table. Used for auditing, validation, cascading changes. Can be BEFORE/AFTER or INSTEAD OF.
Q99: What is a cursor? +
A cursor iterates through query result sets row by row. Used when row-level processing is needed. Types: FORWARD_ONLY, SCROLL. Slower than set-based operations.
Q100: What is the difference between a clustered and non-clustered index? +
Clustered: determines physical order of data, one per table, leaf nodes = actual data. Non-clustered: separate structure, up to 999 per table, leaf nodes = pointers to data.
Q101: What is a primary key vs unique key constraint? +
PK: one per table, NOT NULL, enforces entity integrity. UNIQUE: multiple allowed, allows one NULL, ensures no duplicates. Both create indexes.
Q102: What is a transaction? +
A logical unit of work with ACID properties. All SQL operations succeed (COMMIT) or all are rolled back (ROLLBACK). Ensures data consistency.
Q103: What is a deadlock? +
Deadlock occurs when two or more transactions hold locks on resources each other needs, causing them to wait indefinitely. DBMS resolves by killing one transaction (deadlock victim).
Q104: What is normalization? +
The process of organizing data to minimize redundancy and dependency. Divides large tables into smaller, related tables. Normal forms: 1NF, 2NF, 3NF, BCNF.
Q105: What is denormalization? +
Deliberately adding redundancy to improve query performance. Opposite of normalization. Used in data warehousing, read-heavy systems. Trade-off: faster reads, slower writes, more storage.
Q106: What is ACID in databases? +
Atomicity (all-or-nothing), Consistency (valid state), Isolation (concurrent transactions don't interfere), Durability (committed changes persist).
Q107: Explain the COMMIT and ROLLBACK commands. +
COMMIT saves all changes permanently. ROLLBACK undoes changes since last COMMIT. Both end the current transaction. ROLLBACK can also go to a SAVEPOINT.
Q108: What is a SAVEPOINT? +
SAVEPOINT creates a point within a transaction that you can roll back to without affecting the entire transaction. Useful for partial rollbacks.
Q109: What is the difference between an OLTP and OLAP system? +
OLTP: online transaction processing (high volume, simple queries, normalized, real-time). OLAP: online analytical processing (complex queries, aggregations, historical data, denormalized).
Q110: What is a database schema? +
A schema is the logical container of database objects (tables, views, indexes, etc.). Different users can have separate schemas. Provides organization and access control.
Q111: What is an alias in SQL? +
An alias is a temporary name for a table or column within a query. Improves readability. Table alias: FROM emp e. Column alias: SELECT salary AS sal.
Q112: What is the difference between EXISTS and IN operators? +
EXISTS checks if a subquery returns any rows (stops at first match, faster for large sets). IN checks if value matches any in a list. EXISTS is more efficient for correlated subqueries.
Q113: What is a CASE statement in SQL? +
CASE is a conditional expression like IF-THEN-ELSE. Two forms: Simple (CASE col WHEN val THEN result) and Searched (CASE WHEN condition THEN result). Used in SELECT, WHERE, ORDER BY.
Q114: What is a function vs stored procedure? +
Function returns a single value, can be used in SELECT. Stored procedure may return multiple results, can modify data, supports transactions. Functions are pure (can't modify DB state).
Q115: What are string functions in SQL? +
Common: CONCAT, LENGTH, UPPER/LOWER, SUBSTRING, TRIM, REPLACE, INSTR. Used for string manipulation in queries.
Q116: What are date functions in SQL? +
Common: NOW, CURDATE, DATE_FORMAT, DATE_ADD, DATEDIFF, YEAR/MONTH/DAY. Used for date arithmetic and formatting.
Q117: What is the COALESCE function? +
COALESCE returns the first non-NULL value from a list of expressions. Useful for handling NULLs: COALESCE(col, default_value).
Q118: What is the difference between NULL and empty string? +
NULL means 'no value' / unknown. Empty string '' is a valid value (zero-length string). NULL comparisons use IS NULL, not =. Aggregate functions ignore NULL but count empty strings.

🧮 Relational Algebra

Q119: What is relational algebra? +
Relational algebra is a procedural query language for the relational model. Operations take relations as input and produce new relations. Foundation of SQL query processing.
Q120: What are the fundamental operations of relational algebra? +
SELECT (σ), PROJECT (π), UNION (∪), SET DIFFERENCE (-), CARTESIAN PRODUCT (×), RENAME (ρ). These form the complete set of basic operations.
Q121: What is the SELECT operation (σ) in relational algebra? +
SELECT (σ) filters rows based on a condition. Notation: σ_condition(R). Example: σ_age>18(Person) returns rows with age > 18 from Person relation.
Q122: What is the PROJECT operation (π) in relational algebra? +
PROJECT (π) selects specific columns. Notation: π_column1,column2(R). Eliminates duplicates. Example: π_name,age(Student) returns only name and age columns.
Q123: Explain the UNION operation (∪) in relational algebra. +
UNION combines two compatible relations (same attributes, same domains). Returns all tuples from both, removing duplicates. Notation: R ∪ S.
Q124: What is the SET DIFFERENCE operation (-) in relational algebra? +
Returns tuples in first relation but not in second. Both must be compatible. Notation: R - S. Equivalent to MINUS in SQL.
Q125: What is the CARTESIAN PRODUCT (×) operation? +
CARTESIAN PRODUCT combines every tuple of R with every tuple of S. Result has all attributes from both. Notation: R × S. Usually followed by SELECT for meaningful results.
Q126: Explain the JOIN operation (⋈) in relational algebra. +
JOIN combines related tuples from two relations. Theta join: R ⋈_condition S. Equi-join: condition uses equality. Natural join: equi-join on all common attributes, removes duplicate columns.
Q127: What is the RENAME operation (ρ) in relational algebra? +
RENAME gives a new name to a relation or attribute. Notation: ρ_new_name(R) or ρ_new_attr/old_attr(R). Essential for self-joins and avoiding ambiguity.
Q128: What is an equi-join in relational algebra? +
An equi-join is a theta-join with an equality condition. Example: R ⋈_{R.id = S.id} S. Produces result with two identical join columns (unless projected).
Q129: What is a natural join in relational algebra? +
Natural join (⋈) is an equi-join on all attributes with same name, with duplicate columns removed. Example: R ⋈ S automatically joins on all common attribute names.
Q130: Explain the OUTER JOIN in relational algebra. +
Outer join preserves non-matching tuples by padding with NULLs. Left outer join (⟕): preserves all left tuples. Right outer join (⟖): preserves all right. Full outer join (⟗): preserves all from both.
Q131: What is the DIVISION operation (÷) in relational algebra? +
Division (R ÷ S) finds tuples in R related to ALL tuples in S. Used for 'all' queries. Example: Find students who take ALL courses. Complex operation expressible using basic operations.
Q132: What is the SEMIJOIN operation (⋉)? +
Semijoin returns tuples from left relation that have a matching tuple in right relation. Like inner join but only returns left attributes. R ⋉ S = π_attributes(R)(R ⋈ S).
Q133: How do you express INTERSECTION in relational algebra? +
INTERSECTION (R ∩ S) = R - (R - S). Returns tuples present in both relations. Requires compatible relations.
Q134: What is an outer union in relational algebra? +
Outer union handles relations with different attributes. Missing attributes padded with NULLs. Not part of standard relational algebra but useful in practice.
Q135: What is the difference between UNION and INTERSECTION? +
UNION returns all tuples from both relations (union). INTERSECTION returns only tuples present in both (overlap). Both require compatible relations.
Q136: Explain the purpose of the RENAME operation. +
RENAME (ρ) provides aliases for relations and attributes. Essential for: self-joins, avoiding name conflicts, making expressions readable.
Q137: How do you perform set difference in relational algebra? +
R - S returns all tuples in R not in S. Both must be union-compatible (same attributes and domains). Equivalent to NOT IN or NOT EXISTS in SQL.
Q138: What is the DIVISION operation, and when is it used? +
R ÷ S returns tuples in R that relate to ALL tuples in S. Used for universal quantification queries. Example: 'Students who have taken all courses'. No direct SQL equivalent (uses NOT EXISTS).
Q139: Describe the SEMIJOIN operation and its significance. +
Semijoin (R ⋉ S) returns rows from R that have at least one matching row in S. Significant for query optimization: can reduce rows before expensive joins.
Q140: What is the NATURAL JOIN operation? +
Natural join automatically joins on all attributes with same name, removes duplicate columns. Example: Student ⋈ Enrolls joins on student_id. Simpler syntax but less control.
Q141: Explain the purpose of the OUTER JOIN operation. +
Outer join preserves non-matching tuples. Useful when you need all data from one relation regardless of match. Left/Right/Full variants for different preservation needs.
Q142: How do you perform aggregation in relational algebra? +
Extended relational algebra adds aggregation (𝒢). Notation: 𝒢_{count(*), avg(salary)}(Employee). GROUP BY: 𝒢_{dept_id, count(*)}(Employee).
Q143: Describe how recursive queries can be represented in relational algebra. +
Recursive queries (e.g., org charts, bill of materials) require transitive closure. Extended relational algebra adds recursion (R = R ∪ expression). Not expressible in pure relational algebra.
Q144: How can you optimize relational algebra expressions? +
Push SELECT down (filter early), combine cascading SELECTs, use efficient JOIN order, choose appropriate JOIN types, replace × with ⋈, early PROJECT to reduce columns.
Q145: Explain the concept of query optimization in relational algebra. +
Query optimization transforms expressions into equivalent but more efficient ones. Uses equivalence rules to minimize intermediate result sizes and processing costs.
Q146: What are some common optimization techniques used in relational algebra? +
Selection pushdown, projection pushdown, join reordering, pipelining, using indexes, replacing expensive operations (×) with joins, merging cascading operations.
Q147: How does indexing impact query performance in relational algebra? +
Indexes provide fast access paths for SELECT conditions. Can reduce full table scans to index lookups. Query optimizer uses indexes when beneficial.
Q148: Describe the concept of query execution plans in relational algebra. +
Query execution plan is a sequence of relational algebra operations the DBMS executes. Includes access methods, join algorithms, sort operations. Optimizer chooses lowest-cost plan.
Q149: Discuss the role of relational algebra in relational database theory. +
Relational algebra provides theoretical foundation for relational model. Defines query semantics formally, enables query optimization, proves equivalence of SQL queries.
Q150: How does relational algebra facilitate database query processing? +
SQL queries are translated to relational algebra trees. Optimizer transforms these trees using equivalence rules. Execution engine evaluates the optimized plan.
Q151: Explain how relational algebra operations are used in practice. +
SELECT = filter, PROJECT = column selection, JOIN = combining tables, UNION = combining results. SQL internally compiles to relational algebra expressions.
Q152: Describe the relationship between relational algebra and database normalization. +
Both based on relational theory. Relational algebra operates on normalized relations. FDs determine normalization. Normalization ensures relational algebra operations produce correct results.
Q153: How do you write a relational algebra expression for Cartesian product? +
R × S. Combines every row of R with every row of S. Result has degree = degree(R) + degree(S), cardinality = cardinality(R) × cardinality(S).
Q154: Explain how you would retrieve data from multiple tables using relational algebra. +
Use JOIN operations: π_columns(R ⋈_condition S). Or Cartesian product + SELECT: π_columns(σ_condition(R × S)).
Q155: How do you perform nested queries in relational algebra? +
Nested queries use sub-expressions. Example (division): R ÷ S. Correlated queries use renaming to create copies of relations.
Q156: Provide an example of a complex query expressed in relational algebra. +
Find employees earning more than average: π_name(Employee ⋈_{emp.salary > avg.salary} (ρ_avg(dept_id, avg_salary)(𝒢_{dept_id, AVG(salary)→avg_salary}(Emp)))).
Q157: Can you provide examples of real-world scenarios where relational algebra is used? +
Query optimization in DBMS internals, data integration, ETL processing, query language design, database theory research, SQL compliance testing.
Q158: Describe the process of composing complex relational algebra expressions. +
Start with base relations, apply unary operations (σ, π), combine with binary operations (⋈, ∪), rename for clarity, use temporary results for readability.
Q159: How does relational algebra relate to the relational model of data? +
Relational algebra is the query language for the relational model. The model defines data structure (relations); algebra defines operations on that structure.
Q160: How does relational algebra support data manipulation in DBMS? +
Provides formal framework for: retrieval (σ, π, ⋈), set operations (∪, ∩, −), updates (insert = ∪, delete = −). Foundation for SQL implementation.
Q161: What is the purpose of transaction management in a DBMS? +
Transaction management ensures ACID properties, handles concurrency, manages recovery after failures, and provides isolation between concurrent transactions.
Q162: What are the different states that a transaction can be in during its execution? +
Active (executing), Partially Committed (all operations done, in buffer), Committed (changes persisted), Failed (error occurred), Aborted (rolled back), Terminated (finished).
Q163: How does a DBMS ensure that transactions are executed reliably? +
Through ACID properties, logging (undo/redo), recovery protocols, concurrency control, lock management, and checkpoint mechanisms.
Q164: Describe the role of locks and latches in transaction management. +
Locks protect database logical content (tables, rows) for transaction isolation. Latches protect internal data structures (buffers, indexes) for short durations. Locks are held for transaction duration; latches are brief.

⚙️ Transaction Management

Q165: What is a deadlock, and how does a DBMS handle deadlock situations? +
Deadlock: two+ transactions wait indefinitely for each other's locks. Handling: prevention (acquire all locks upfront), avoidance (wait-die, wound-wait), detection (wait-for graph), recovery (kill victim).
Q166: How do you handle transaction rollback and commit operations? +
COMMIT: write log records to stable storage, release locks. ROLLBACK: use undo log to revert changes, release locks. Partial rollback via SAVEPOINT.
Q167: What is concurrency control, and why is it important? +
Concurrency control manages simultaneous transaction execution to ensure isolation. Prevents: lost updates, dirty reads, non-repeatable reads, phantom reads. Essential for data consistency.
Q168: Explain the difference between pessimistic and optimistic concurrency control. +
Pessimistic: locks resources upfront (assumes conflicts likely). Optimistic: executes without locks, validates at commit (assumes conflicts rare). Optimistic better for read-heavy workloads.
Q169: How do you implement locking mechanisms for concurrency control? +
Shared locks (S) for reads, Exclusive locks (X) for writes. Lock table tracks held locks. Lock manager grants/queues lock requests. Deadlock detection/ prevention required.
Q170: Describe the concept of two-phase locking (2PL) and its role. +
2PL: transactions acquire all locks in growing phase, release in shrinking phase. Ensures serializability. Variants: Strict 2PL (release after commit), Rigorous 2PL (release at commit).
Q171: What is a transaction isolation level, and how does it affect concurrency? +
Defines how transaction changes are visible to others. Higher isolation = more consistency but less concurrency. Levels: READ UNCOMMITTED (lowest), READ COMMITTED, REPEATABLE READ, SERIALIZABLE (highest).
Q172: Explain the different isolation levels (READ UNCOMMITTED, READ COMMITTED, REPEATABLE READ, SERIALIZABLE). +
READ UNCOMMITTED: dirty reads allowed. READ COMMITTED: only committed data (no dirty reads). REPEATABLE READ: same read returns same result (no non-repeatable reads). SERIALIZABLE: phantom-proof, highest.
Q173: What are the trade-offs between isolation levels? +
Higher isolation = more locking = less concurrency = slower. Lower isolation = faster but more anomalies. Choose based on consistency requirements vs performance needs.
Q174: How does a DBMS ensure the isolation property for each isolation level? +
READ UNCOMMITTED: no locks. READ COMMITTED: read locks released immediately. REPEATABLE READ: read locks held until commit. SERIALIZABLE: range locks + read locks until commit.
Q175: Can you provide examples of scenarios where different isolation levels would be appropriate? +
READ UNCOMMITTED: approximate counts. READ COMMITTED: typical web apps. REPEATABLE READ: financial reporting. SERIALIZABLE: banking transfers, critical transactions.
Q176: Describe the process of transaction recovery in a DBMS. +
After crash: analyze log to determine active/committed/aborted transactions. Redo committed transactions (forward). Undo aborted/incomplete transactions (backward). Use checkpoints to limit recovery work.
Q177: What is a checkpoint, and how does it help in transaction recovery? +
Checkpoint: periodically writes dirty buffers and log records to disk. Limits recovery scope — only transactions after last checkpoint need recovery. Reduces restart time.
Q178: Explain the role of the transaction log in recovery operations. +
Log records all DB modifications (before/after images) in stable storage. Used for: UNDO (revert failed transactions), REDO (replay committed transactions). Essential for durability.
Q179: How does a DBMS handle system crashes during transaction execution? +
Crash recovery: analyze log → REDO all committed transactions → UNDO all uncommitted transactions. Database returns to consistent state. ACID durability ensures committed data survives.
Q180: Explain the difference between forward and backward recovery techniques. +
Forward recovery (REDO): reapply changes from log. Used for committed transactions. Backward recovery (UNDO): revert changes using before-images. Used for aborted transactions.
Q181: What are distributed transactions, and how do they differ from local transactions? +
Distributed transactions span multiple DB nodes/servers. Need distributed commit protocol (2PC). Challenges: network failures, coordination, consistency. Local transactions run on single node.
Q182: Describe the challenges associated with managing distributed transactions. +
Network latency, partial failures, distributed deadlock detection, clock synchronization, data consistency maintenance (CAP theorem), two-phase commit complexity.
Q183: How do you ensure atomicity and consistency in distributed transactions? +
Two-Phase Commit (2PC): coordinator asks all participants to prepare (phase 1), then commits/aborts all (phase 2). Ensures all-or-nothing across nodes.
Q184: Explain the role of distributed locks in ensuring data integrity across multiple sites. +
Distributed locks coordinate access to shared resources across nodes. Centralized lock manager or distributed algorithm (e.g., Lamport's bakery). Prevent concurrent conflicting updates.
Q185: What is a two-phase commit protocol, and how does it work? +
Phase 1 (Prepare): coordinator asks all participants to prepare (vote yes/no). Phase 2 (Commit/Abort): if all yes, commit; if any no, abort. Ensures atomic distributed commit.
Q186: How can you optimize transaction processing for better performance? +
Use appropriate isolation levels, batch transactions, optimize queries, use indexes, connection pooling, reduce lock contention, use optimistic concurrency where possible.
Q187: Describe the concept of transaction batching and its impact on performance. +
Batching groups multiple operations into one transaction. Reduces commit overhead, log writes, network round trips. Improves throughput but may increase latency for individual operations.
Q188: What are the factors that affect transaction throughput and response time? +
Lock contention, I/O speed, log writes, network latency, query complexity, concurrency level, isolation level, hardware resources (CPU, RAM, disk).
Q189: Explain the role of indexing and caching in improving transaction performance. +
Indexes speed up data access (reduce I/O). Caching keeps frequently accessed data in memory (reduce disk reads). Both significantly reduce transaction latency.
Q190: How do you measure and monitor transaction performance in a DBMS? +
Metrics: throughput (TPS), latency, lock wait times, deadlock counts, log write times, buffer cache hit ratio, query execution times. Tools: DBMS monitoring views, perf monitors.
Q191: Can you provide examples of real-world scenarios where transaction management is crucial? +
Banking (fund transfers), e-commerce (order placement), airline bookings (seat reservation), inventory management (stock updates), healthcare (patient records).
Q192: How do e-commerce platforms utilize transactions to ensure data consistency? +
Order transactions: reduce inventory + charge payment + create order atomically. If any step fails, entire transaction rolls back. Ensures no overselling or incomplete orders.
Q193: Describe how banking systems handle transactions to maintain accurate balances. +
Transfer transaction: debit source + credit destination atomically. Uses SERIALIZABLE isolation. Audit logs track every change. If system crashes mid-transfer, recovery ensures consistency.
Q194: What are some challenges unique to handling transactions in large-scale distributed systems? +
Network partitions (CAP), clock skew, partial failures, cross-datacenter latency, coordination overhead, deadlock detection across nodes, maintaining strong consistency at scale.
Q195: How do social media platforms ensure data consistency in transactional operations? +
Often use eventual consistency (BASE) for non-critical data. Use strong consistency for critical operations (payments, account settings). Employ conflict resolution strategies (CRDTs, last-write-wins).

🔗 Functional Dependencies

Q196: What is a functional dependency (FD)? +
X → Y means X uniquely determines Y. For any two rows with same X value, Y values must be equal. Used to define keys and normalization.
Q197: What is Armstrong's Axioms for functional dependencies? +
Reflexivity: if β ⊆ α then α → β. Augmentation: if α → β then αγ → βγ. Transitivity: if α → β and β → γ then α → γ. These are sound and complete.
Q198: What are the additional inference rules derived from Armstrong's Axioms? +
Union: α → β and α → γ ⇒ α → βγ. Decomposition: α → βγ ⇒ α → β and α → γ. Pseudotransitivity: α → β and βγ → δ ⇒ αγ → δ.
Q199: What is the closure of a set of functional dependencies? +
Closure F+ = all FDs logically implied by F. Computed by repeatedly applying Armstrong's axioms until no new FDs can be derived.
Q200: What is the closure of an attribute set? +
Attribute closure α+ = set of all attributes functionally determined by α. Used to: determine if α is super key (α+ = all attributes), check FD membership.
Q201: What is a canonical cover of FDs? +
Canonical cover Fc is a minimal set of FDs equivalent to F. No redundant FDs, no extraneous attributes, LHS unique. Used for normalization and schema design.
Q202: What are the different types of functional dependencies? +
Trivial (β ⊆ α): always true. Non-trivial: β ⊈ α. Full: β depends on entire α. Partial: β depends on subset of α. Transitive: α → β and β → γ.
Q203: How do functional dependencies relate to normalization? +
Each normal form addresses specific FD issues: 2NF removes partial FDs, 3NF removes transitive FDs, BCNF ensures all determinants are super keys. FDs define the normalization process.
Q204: What is a multivalued dependency (MVD)? +
X →→ Y means X determines multiple independent values of Y. Not captured by FDs. Addressed by 4NF. Example: Employee →→ Phone (one employee has multiple phones).
Q205: What is the difference between a functional dependency and a multivalued dependency? +
FD: one-to-one relationship (X → Y: one X value → one Y value). MVD: one-to-many (X →→ Y: one X value → multiple independent Y values).
Q206: What is a lossless join decomposition? +
Splitting a relation R into R1 and R2 such that R1 ⋈ R2 = R (no spurious tuples). Guaranteed if common attributes form a super key in at least one of R1 or R2.
Q207: What is dependency preservation? +
After decomposition, all original FDs can be checked on individual decomposed relations without needing joins. Ensures constraints can be enforced locally.
Q208: How does normalization affect query performance? +
More tables → more joins → slower reads. But faster writes (less redundancy). Trade-off between read and write performance. Denormalization used for read-heavy workloads.

📦 Unstructured Databases

Q209: What is an unstructured database? +
A database that stores unstructured data (text, images, videos, JSON, documents) without a fixed schema. Unlike relational databases, doesn't require predefined table structures.
Q210: How does an unstructured database differ from a structured database? +
Structured: fixed schema, tables, rows/columns, SQL. Unstructured: flexible/ schema-less, various data formats, NoSQL querying, better horizontal scaling.
Q211: Can you provide examples of unstructured data formats? +
Text documents (PDF, DOCX), images (JPEG, PNG), videos (MP4), audio (MP3), JSON/XML files, emails, social media posts, log files, sensor data.
Q212: Explain the challenges of managing and querying unstructured data. +
No standard schema, complex querying (needs full-text search, content-based retrieval), variable formats, storage management, lack of standard APIs, data quality issues.
Q213: How do unstructured databases handle schema flexibility and schema evolution? +
Schema-on-read: structure is applied at query time, not write time. Documents can have different fields. New fields added automatically without migrations.
Q214: Describe the storage mechanism used in unstructured databases. +
Object storage (S3, blob), document storage (JSON/BSON), key-value pairs, column families, or graph structures. Often uses distributed storage with replication.
Q215: How do you efficiently retrieve and query unstructured data? +
Indexing (full-text, inverted indexes), metadata tagging, content-based search, MapReduce, parallel processing. Tools: ElasticSearch for text, image recognition for media.
Q216: Explain the role of indexing in unstructured databases. +
Indexes enable fast search over unstructured content. Types: inverted indexes (text), spatial indexes (geo), vector indexes (embeddings). Critical for query performance.
Q217: What are the trade-offs between storage and retrieval techniques for unstructured data? +
More indexes = faster reads, slower writes, more storage. Compression saves space but costs CPU. Replication improves read performance but consumes storage.
Q218: What are common methods for querying unstructured databases? +
Full-text search, regex patterns, JSON path queries, MapReduce, aggregation pipelines, vector similarity search, graph traversal.
Q219: Explain the concept of full-text search and how it's implemented. +
Full-text search indexes every word in text documents. Uses inverted indexes mapping words to documents. Supports ranking (TF-IDF, BM25), stemming, fuzzy matching, relevance scoring.
Q220: How do you perform advanced analytics on unstructured data? +
NLP for text, computer vision for images, speech-to-text for audio, sentiment analysis, topic modeling, entity extraction. Tools: Hadoop, Spark, ML frameworks.
Q221: Describe the challenges of complex queries on unstructured vs structured data. +
Structured: well-defined operations (SQL joins, aggregations). Unstructured: lacks standard query model, needs special indexing, approximate results, more complex optimization.
Q222: How do unstructured databases scale horizontally? +
Distribute data across nodes using sharding (consistent hashing). Auto-rebalance when nodes added/removed. Examples: MongoDB sharding, Cassandra ring topology.
Q223: What are strategies for optimizing performance in unstructured databases? +
Proper indexing, data partitioning, caching (Redis), denormalization, materialized views, read replicas, compression, query optimization.
Q224: Explain how distributed computing improves scalability in unstructured databases. +
Data partitioned across nodes, parallel query execution, MapReduce for batch processing, replication for availability. Linear scalability by adding nodes.
Q225: Discuss the role of caching and replication in unstructured databases. +
Caching: store hot data in memory (Redis, Memcached) for low latency. Replication: copy data across nodes for read scaling and fault tolerance. Both improve performance and availability.
Q226: How do you integrate unstructured data from different sources? +
ETL pipelines, data lakes, schema-on-read, data virtualization, APIs, message queues. Normalize into common format (JSON, Parquet) for analysis.
Q227: Describe the process of transforming unstructured data into structured format. +
Parse raw data → extract fields → apply schema → validate → load into structured storage. Examples: NLP extracts entities from text, OCR extracts text from images.
Q228: What are challenges of integrating and transforming unstructured data? +
Data variety, inconsistent formats, missing metadata, scale, real-time requirements, data quality, schema mapping complexity, computational cost.
Q229: How do you ensure data security in unstructured databases? +
Encryption at rest and in transit, access control (RBAC), audit logging, data masking, secure deletion, network isolation, regular security assessments.
Q230: What are common security vulnerabilities in unstructured databases? +
No authentication by default, injection attacks (NoSQL injection), insecure APIs, misconfigured access controls, unencrypted data, lack of auditing.
Q231: Explain the role of access control in protecting unstructured data. +
Authentication (who you are), Authorization (what you can do), Auditing (what you did). Role-based access (RBAC), attribute-based (ABAC). Policies at document/field level.
Q232: How do unstructured databases handle schema evolution? +
Schema flexibility allows fields to be added/removed per document. Versioning: maintain multiple schema versions, handle backward compatibility in application code.
Q233: Discuss challenges of maintaining backward compatibility with schema changes. +
Old documents may lack new fields, new code must handle missing fields gracefully. Strategy: default values, null checks, gradual migration, dual-write during transition.
Q234: What strategies manage schema changes with minimal disruption? +
Additive changes (new fields), gradual migration, read-time schema resolution, versioned APIs, feature flags, rolling upgrades, backward-compatible changes.
Q235: Can you provide examples of industries using unstructured databases? +
Healthcare (medical records, images), Media (content storage), E-commerce (product catalogs), Social media (user content), Finance (documents, communications), Scientific research.
Q236: How do content management systems use unstructured databases? +
Store documents, images, videos as objects. Use metadata for search. Versioning for content history. CDN integration for delivery. Examples: WordPress, Drupal, AEM.
Q237: Explain how social media platforms use unstructured databases. +
Store posts, comments, images, videos in NoSQL databases (MongoDB, Cassandra). Use graph DBs for social connections. Real-time feeds using message queues.
Q238: What challenges in healthcare can be addressed using unstructured databases? +
Medical image storage (MRI, X-ray), patient records, genomic data, research papers, clinical notes. Need HIPAA compliance, scalable storage, fast retrieval.
Q239: Discuss the role of unstructured databases in NLP and sentiment analysis. +
Store large text corpora for training ML models. Process and analyze unstructured text. Support vector search for semantic similarity. NLP pipelines extract insights.
Q240: What emerging technologies are shaping the future of unstructured databases? +
Vector databases (Pinecone, Milvus), AI-powered search, serverless databases, multi-model databases, edge computing, automated schema inference.
Q241: How do ML and AI impact unstructured databases? +
Enable intelligent search (semantic, image), automatic tagging, content classification, anomaly detection, recommendation systems. AI models process unstructured content.
Q242: Discuss the potential impact of blockchain on unstructured databases. +
Decentralized storage (IPFS, Filecoin), immutable audit trails, data provenance, smart contracts for access control. Challenges: scalability, latency, cost.
Q243: What are key areas of research in unstructured databases? +
Vector search optimization, AI-native databases, automated indexing, hybrid transactional-analytical processing (HTAP), multi-model integration, edge-to-cloud data management.
Q244: What are the ACID properties in detail? +
Atomicity: transaction is all-or-nothing. Consistency: transaction takes DB from consistent state to consistent state. Isolation: concurrent execution appears serial. Durability: committed changes survive failures.
Q245: Explain the concept of serializability in transactions. +
A schedule is serializable if it produces the same result as some serial execution of the same transactions. Conflict serializable (conflict graph acyclic) and view serializable.
Q246: What is a schedule in transaction processing? +
A schedule is the order of operations from multiple transactions as they execute concurrently. Types: serial (no concurrency), serializable (equivalent to serial), non-serializable.
Q247: What is a conflict in transaction schedules? +
Two operations conflict if they belong to different transactions, access same data item, and at least one is a write. Conflicts determine serializability.
Q248: What is a conflict serializability test? +
Build a precedence graph: nodes = transactions, edges = conflicts. If graph has a cycle, schedule is not conflict serializable. Acyclic implies conflict serializable.
Q249: What is the difference between conflict serializable and view serializable? +
Conflict serializable: can be transformed to serial by swapping non-conflicting operations. View serializable: produces same result as some serial schedule. View serializable is broader.
Q250: What is a timestamp-based concurrency control protocol? +
Each transaction gets a unique timestamp. Operations ordered by timestamp: older transactions have priority. Uses Thomas Write Rule. No locks needed but may cause cascading aborts.
Q251: What is multiversion concurrency control (MVCC)? +
DBMS maintains multiple versions of each data item. Readers see snapshot (old version), writers create new version. Provides snapshot isolation. Used by PostgreSQL, Oracle, MySQL InnoDB.
Q252: What is snapshot isolation? +
Each transaction sees a snapshot of committed data at its start time. Writes go to new version. Detects write-skew anomalies. Not serializable but widely used for performance.
Q253: What is a write-ahead log (WAL)? +
Log records written to stable storage BEFORE actual data changes. Ensures atomicity (undo) and durability (redo). Fundamental to ARIES recovery algorithm.
Q254: What is the ARIES recovery algorithm? +
ARIES: Algorithms for Recovery and Isolation Exploiting Semantics. Three phases: Analysis (find dirty pages/active transactions), REDO (reapply from last checkpoint), UNDO (rollback losers).
Q255: What is a buffer pool in a DBMS? +
Memory area caching database pages. Reduces disk I/O. Uses replacement policies (LRU, Clock). Dirty pages written by background writer. Managed by buffer manager.
Q256: What is the difference between row-level and table-level locking? +
Row-level: lock individual rows (high concurrency, more overhead). Table-level: lock entire table (low concurrency, less overhead). InnoDB uses row-level; MyISAM uses table-level.
Q257: What is intention locking? +
Hierarchical locking protocol. Intention-shared (IS): intends to lock lower-level objects in shared mode. Intention-exclusive (IX): intends exclusive. Shared-intention-exclusive (SIX): shared + intention-exclusive.
Q258: What is a lock escalation? +
Automatically converting many fine-grained locks (row-level) into one coarse lock (table-level) to reduce overhead. Usually triggered when lock count exceeds threshold.
Q259: What is a distributed database? +
A database with data stored across multiple physical locations (servers/sites). Managed by a distributed DBMS. Advantages: reliability, availability, local autonomy.
Q260: What are the types of distributed database systems? +
Homogeneous: all sites use same DBMS software. Heterogeneous: sites may use different DBMS, need middleware for integration. Federated: autonomous databases with partial integration.
Q261: What is data fragmentation in distributed databases? +
Splitting data across sites. Horizontal: row-wise splitting. Vertical: column-wise splitting. Mixed: both. Used to keep data close to its users.
Q262: What is data replication in distributed databases? +
Maintaining copies of data at multiple sites. Types: synchronous (immediate update all copies) and asynchronous (eventual consistency). Benefits: availability, read performance.
Q263: What is the CAP theorem? +
In a distributed system, you can have at most 2 of 3: Consistency (all nodes same data), Availability (every request gets response), Partition tolerance (works despite network splits).
Q264: What is the PACELC theorem? +
Extension of CAP: if Partition occurs (P), choose Availability or Consistency (AC). Else (E), choose Latency or Consistency (LC). More practical trade-off framework.
Q265: What is BASE in NoSQL databases? +
Basically Available (system always available), Soft state (state may change over time), Eventual consistency (will become consistent eventually). Contrasts with ACID.
Q266: What is eventual consistency? +
Given enough time without updates, all replicas will converge to the same value. Used in DNS, CDNs, NoSQL databases. Sacrifices strong consistency for availability.
Q267: What are CRDTs (Conflict-free Replicated Data Types)? +
Data structures that automatically resolve conflicts without coordination. Used in distributed systems for eventual consistency. Examples: counters, sets, registers.
Q268: What is a vector clock? +
Mechanism to track causal ordering of events in distributed systems. Each node maintains a vector of counters. Used to detect conflicts in distributed databases.
Q269: What is the gossip protocol? +
Decentralized communication: each node periodically exchanges state with random peers. Information spreads exponentially. Used for: failure detection, membership, data propagation.
Q270: What is the difference between synchronous and asynchronous replication? +
Synchronous: transaction commits only after all replicas acknowledge. Strong consistency, higher latency. Asynchronous: transaction commits without waiting for replicas. Lower latency, possible data loss.
