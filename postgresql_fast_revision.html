PostgreSQL Fast Revision Guide

Complete Guide to PostgreSQL Fundamentals - Parts 1 to 3

Part 1: Database & Table Fundamentals
Creating Databases & Schemas

Key Concepts:

  • DATABASE: Top-level container for all your data
  • SCHEMA: Logical grouping within a database (like folders)
  • Hierarchy: Database → Schema → Table → Rows
  • pgcrypto: Extension for UUID generation
-- Create a database (for learning only, not production) DROP DATABASE IF EXISTS postgresql_part1; CREATE DATABASE postgresql_part1; -- Create schema with IF NOT EXISTS to prevent errors CREATE SCHEMA IF NOT EXISTS basics; -- Enable UUID generation CREATE EXTENSION IF NOT EXISTS pgcrypto; -- View all schemas SELECT schema_name FROM information_schema.schemata ORDER BY schema_name;
Creating Tables with Data Types

Essential Data Types:

  • SERIAL: Auto-incrementing integer (1, 2, 3...)
  • TEXT: Variable-length string
  • VARCHAR(n): String with max length
  • INTEGER: Whole numbers
  • BIGINT: Larger whole numbers
  • NUMERIC(p,s): Exact decimal (precision, scale)
  • BOOLEAN: TRUE/FALSE values
  • TIMESTAMP: Date and time
  • UUID: Universal unique identifier
  • JSONB: Binary JSON data
-- Comprehensive table with multiple data types CREATE TABLE basics.students ( id SERIAL PRIMARY KEY, name TEXT NOT NULL, email TEXT NOT NULL UNIQUE, age INTEGER CHECK (age >= 18), created_at TIMESTAMP DEFAULT NOW() ); -- Products table with various types CREATE TABLE basics.products_basic ( id SERIAL PRIMARY KEY, name VARCHAR(100) NOT NULL, description TEXT, stock INTEGER DEFAULT 0, total_views BIGINT DEFAULT 0, price NUMERIC(10,2), is_active BOOLEAN DEFAULT true ); -- UUID and JSONB example CREATE TABLE basics.app_events ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), event_name TEXT NOT NULL, metadata JSONB DEFAULT '{}'::jsonb, created_at TIMESTAMP DEFAULT NOW() );
Constraints: The Rules of Data

Constraint Types:

  • NOT NULL: Column cannot be empty
  • UNIQUE: No duplicate values allowed
  • DEFAULT: Automatic value if none provided
  • CHECK: Validates data meets specific conditions
  • PRIMARY KEY: Unique identifier (NOT NULL + UNIQUE)
CREATE TABLE basics.accounts ( id SERIAL PRIMARY KEY, full_name TEXT NOT NULL, email TEXT NOT NULL UNIQUE, is_active BOOLEAN DEFAULT true, age INTEGER CHECK (age >= 18), created_at TIMESTAMP DEFAULT NOW() ); -- INSERT with constraints INSERT INTO basics.accounts (full_name, email, age) VALUES ('sangam mukherjee', 'sangam@gmail.com', 20);
NULL vs Empty String vs Zero

Understanding the Differences:

  • NULL: Unknown/missing value
  • Empty String (''): Known value with no characters
  • Zero (0): Actual numeric value
  • Important: Use IS NULL, not = NULL
SELECT * FROM basics.value_examples WHERE nickname IS NULL; SELECT * FROM basics.value_examples WHERE score = 0; SELECT * FROM basics.value_examples WHERE nickname IS NOT NULL;
JSONB Operators & Queries

JSONB Operators:

  • ->> Get value as text by key
  • -> Get value by key (preserves type)
  • ? Check if key exists
  • @> Check if JSON contains
-- Insert JSONB data INSERT INTO basics.app_events (event_name, metadata) VALUES ('sign_up', '{"browser": "chrome"}'), ('sign_in', '{"user": "sangam"}'); -- Extract value as text using ->> -- Check if key exists using ? SELECT event_name, metadata ->> 'browser' AS browser FROM basics.app_events WHERE metadata ? 'browser';
Part 2: CRUD Operations & Filtering
INSERT: Adding Data

Insert Methods:

  • Single Row: One record at a time
  • Multiple Rows: Batch insert for efficiency
  • RETURNING: Get data back after insert
-- Single row insert INSERT INTO products (name, category, price, stock, sku, description) VALUES ('Laptop stand', 'electronics', 5000.00, 23, 'ELEC-KEY-003', 'laptop stand description'); -- Multiple rows insert INSERT INTO products (name, category, price, stock, sku, description) VALUES ('laptop stand 5', 'electronics', 100, 45, 'ELEC-LAP-100', 'laptop stand 5 desc'), ('laptop stand 6', 'electronics', 100, 45, 'ELEC-LAP-101', 'laptop stand 6 desc'), ('laptop stand 7', 'electronics', 100, 45, 'ELEC-LAP-102', 'laptop stand 7 desc'); -- Insert with RETURNING INSERT INTO products (name, category, price, stock, sku, description) VALUES ('webcam', 'electronics', 456.67, 56, 'ELEC-WEB-009', 'webcam description') RETURNING id, name, category, price, stock, created_at;
SELECT: Retrieving Data

Selection Techniques:

  • SELECT *: Get all columns (use sparingly)
  • Specific Columns: Better performance
  • AS (Aliases): Rename output columns
-- Select specific columns with aliases SELECT name AS product_name, price AS selling_price, stock AS available_quantity FROM products;
WHERE: Filtering Rows

Filter Operators:

  • =, !=, <>, >, <, >=, <=: Comparison operators
  • AND, OR, NOT: Logical operators
  • IN, NOT IN: Match list of values
  • BETWEEN: Range filtering
  • LIKE, ILIKE: Pattern matching
  • IS NULL, IS NOT NULL: Null checking
-- Basic comparison SELECT name, price FROM products WHERE price > 1000; -- AND & OR with grouping SELECT name, category, price, stock FROM products WHERE is_active = true AND (price < 1000 OR stock >= 100); -- IN operator SELECT name, category, price FROM products WHERE category IN ('electronics', 'furniture'); -- BETWEEN for ranges SELECT name, price FROM products WHERE price BETWEEN 100 AND 2000; -- LIKE/ILIKE for patterns (% = any chars, _ = single char) SELECT name, category, price FROM products WHERE name ILIKE '%desk%'; -- NULL checking SELECT name, description FROM products WHERE description IS NULL;
ORDER BY & LIMIT: Sorting & Pagination

Sorting & Paging:

  • ASC: Ascending order (default)
  • DESC: Descending order
  • LIMIT: Maximum rows to return
  • OFFSET: Rows to skip
  • Pagination Formula: OFFSET = (page - 1) × LIMIT
-- Sort by multiple columns SELECT name, category, price FROM products ORDER BY category ASC, price DESC; -- Pagination example -- Page 1: SELECT name, price FROM products ORDER BY name ASC LIMIT 5 OFFSET 0; -- Page 2: SELECT name, price FROM products ORDER BY name ASC LIMIT 5 OFFSET 5;
UPDATE & DELETE: Modifying Data

Data Modification:

  • UPDATE: Modify existing rows
  • DELETE: Remove rows
  • WHERE: Always use to target specific rows
  • RETURNING: Get affected rows back
-- Update single row UPDATE products SET price = 1199.50, stock = 23 WHERE sku = 'ELEC-KEY-001'; -- Update multiple rows with calculation UPDATE products SET price = ROUND(price * 1.10, 2) WHERE category = 'stationery'; -- Delete with RETURNING DELETE FROM products WHERE sku = 'ELEC-TEMP-013' RETURNING id, name, sku;

💡 Pro Tip: RETURNING Clause

Use RETURNING after INSERT, UPDATE, or DELETE to immediately get the affected rows back. Perfect for logging or confirming changes!

Part 3: Relationships, Joins & Advanced Concepts
Database Design: Tables & Relationships

Relationship Types:

  • One-to-Many: One user → Many posts
  • Many-to-Many: Posts ↔ Tags (via junction table)
  • Foreign Key: Links child table to parent table
-- Users table (Parent) CREATE TABLE users ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), name TEXT NOT NULL ); -- Posts table (Child with Foreign Key) CREATE TABLE posts ( id UUID PRIMARY KEY DEFAULT gen_random_uuid(), user_id UUID NOT NULL REFERENCES users(id), title TEXT NOT NULL, status TEXT NOT NULL DEFAULT 'draft' CHECK (status IN ('draft', 'published')), views INTEGER NOT NULL DEFAULT 0 CHECK (views >= 0) ); -- Junction table for Many-to-Many CREATE TABLE post_tags ( post_id UUID NOT NULL REFERENCES posts(id), tag_id UUID NOT NULL REFERENCES tags(id), PRIMARY KEY (post_id, tag_id) );
JOINs: Combining Tables

INNER JOIN

Returns only matching rows from both tables

SELECT u.name AS author, p.title AS post_title FROM posts p INNER JOIN users u ON p.user_id = u.id WHERE p.status = 'published';

LEFT JOIN

Keeps all rows from left table, matches from right (NULL if no match)

SELECT p.title AS post_title, c.body AS comment_body FROM posts p LEFT JOIN comments c ON p.id = c.post_id ORDER BY p.title;

✅ When to Use Each:

INNER JOIN: When you need only matching records from both tables

LEFT JOIN: When you want all records from the left table, even without matches (e.g., all posts with optional comments)

Table Aliases & Multi-Table Queries
-- Using aliases for cleaner queries SELECT p.title AS post_title, p.status, p.views, u.name AS author_name, c.body AS comment_body FROM posts AS p INNER JOIN users AS u ON p.user_id = u.id LEFT JOIN comments AS c ON p.id = c.post_id ORDER BY p.views DESC; -- Many-to-Many: Posts with Tags SELECT p.title AS post_title, t.name AS tag_name FROM posts p INNER JOIN post_tags pt ON p.id = pt.post_id INNER JOIN tags t ON pt.tag_id = t.id ORDER BY p.title, t.name;
Aggregate Functions

Aggregate Functions:

  • COUNT(*): Number of rows
  • COUNT(DISTINCT): Number of unique values
  • SUM(column): Total value
  • AVG(column): Average value
  • MIN(column): Smallest value
  • MAX(column): Largest value
-- Aggregate statistics SELECT COUNT(*) AS total_posts, MIN(views) AS lowest_views, MAX(views) AS highest_views FROM posts; -- COUNT with FILTER clause SELECT COUNT(*) AS total_posts, COUNT(*) FILTER (WHERE status = 'published') AS published_posts FROM posts;
COUNT(DISTINCT): Counting Unique Values
-- Count unique posts per tag SELECT t.name AS tag_name, COUNT(DISTINCT p.id) AS total_unique_posts FROM tags t LEFT JOIN post_tags pt ON t.id = pt.tag_id LEFT JOIN posts p ON pt.post_id = p.id GROUP BY t.id, t.name ORDER BY total_unique_posts DESC;
GROUP BY & HAVING

Grouping & Filtering Groups:

  • GROUP BY: Groups rows with same values
  • WHERE: Filters rows BEFORE grouping
  • HAVING: Filters groups AFTER grouping
-- Posts per author with filtering SELECT u.name AS author_name, COUNT(p.id) AS total_posts, SUM(p.views) AS total_views FROM users u LEFT JOIN posts p ON u.id = p.user_id GROUP BY u.id, u.name HAVING COUNT(p.id) >= 2 ORDER BY total_posts DESC;
Subqueries
-- Posts performing above average SELECT title, status, views FROM posts WHERE views > ( SELECT AVG(views) FROM posts ) ORDER BY views DESC;
Indexes: Performance Optimization

Index Types:

  • Single Column Index: Index on one column
  • Composite Index: Index on multiple columns
  • Use Case: Speeds up WHERE, JOIN, ORDER BY
-- Single column index CREATE INDEX IF NOT EXISTS idx_posts_status ON posts(status); -- Composite index CREATE INDEX IF NOT EXISTS idx_posts_status_views ON posts(status, views DESC); -- Index for foreign key lookups CREATE INDEX IF NOT EXISTS idx_posts_user_id ON posts(user_id); -- Now queries using these columns are faster! SELECT title, status, views FROM posts WHERE status = 'published' ORDER BY views DESC;
Transactions: Safe Multi-Step Operations

Transaction Commands:

  • BEGIN: Start a transaction
  • COMMIT: Save all changes
  • ROLLBACK: Undo all changes
  • Use Case: Banking, orders, multi-table updates
-- Transaction with COMMIT (success path) BEGIN; UPDATE posts SET status = 'published' WHERE title = 'Indexes for Beginners' AND status = 'draft'; UPDATE posts SET views = views + 50 WHERE title = 'Indexes for Beginners'; SELECT title, status, views FROM posts WHERE title = 'Indexes for Beginners'; COMMIT;
-- Transaction with ROLLBACK (undo changes) BEGIN; -- Accidentally delete all posts! DELETE FROM posts; -- Oops! Let's undo everything ROLLBACK; -- All posts are restored - nothing was committed SELECT * FROM posts;

⚠️ Important:

Transactions ensure ALL operations succeed or ALL fail together. Use ROLLBACK if something goes wrong before COMMIT to undo all changes!
