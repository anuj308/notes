📘 Unit 1 - DevOps Fundamentals

1.1 What is DevOps?

Definition: DevOps is a combination of Development and Operations. It is a culture, set of practices, and tools that integrate and automate the processes between software development and IT operations teams, enabling them to build, test, and release software faster and more reliably.

Core Principles

  • Collaboration: Breaking down silos between Dev and Ops teams
  • Automation: Automating repetitive tasks (builds, tests, deployments)
  • Continuous Integration: Developers frequently merge code changes into a central repository
  • Continuous Delivery: Automating the release process to production
  • Monitoring: Real-time monitoring of applications and infrastructure
  • Infrastructure as Code (IaC): Managing infrastructure through code and version control
🎯 Interview Point: DevOps is NOT a tool or a role — it's a culture and practice that bridges the gap between development and operations.

1.2 Need for DevOps (Traditional vs DevOps)

Problems with Traditional (Waterfall) Approach

  • Long development cycles — months or years between releases
  • Siloed teams — Dev and Ops work independently, causing miscommunication
  • Manual deployments — error-prone and slow
  • Delayed feedback — bugs found late in the cycle
  • Blame culture — "It works on my machine" mentality

How DevOps Solves These

DeploymentManualAutomated
TestingAfter developmentContinuous
Team StructureSiloedCollaborative
Failure RecoveryDays/WeeksMinutes
CommunicationFormal/ScheduledReal-time
📝 Exam Point: DevOps follows the principle of "You build it, you run it" — developers take responsibility for their code in production.

1.5 DevOps Benefits, Disadvantages & Interview Points

✅ Advantages

  • Faster time to market
  • Improved collaboration between teams
  • Reduced failure rates of new releases
  • Faster recovery from failures
  • Automated processes reduce manual errors
  • Better customer satisfaction
  • Scalability and reliability

❌ Disadvantages

  • Steep learning curve for tools and culture shift
  • Initial setup complexity
  • Requires organizational change management
  • Security concerns if not properly integrated (hence DevSecOps)
  • Cost of tooling and training
🎤 Common Interview Q: "What is the difference between Continuous Delivery and Continuous Deployment?"
A: CD (Delivery) = Code is ready to deploy but requires manual approval. CD (Deployment) = Code is automatically deployed to production without human intervention.

🔀 Unit 3 - Git & Version Control

3.1 Version Control System (VCS)

A Version Control System tracks changes to files over time, allowing multiple people to collaborate on the same project.

Why VCS?

  • Track which changes were made, by whom, when, and why
  • Recover any earlier version of the project
  • Work simultaneously without conflicts
  • Store only modified code across versions (saves storage)

3.5 Staging Area & Git Objects

The staging area (index) is where you prepare changes before committing. It allows you to:

  • Preview exactly what will be committed
  • Split work into related commits
  • Record changes before committing to the database

Git Objects (inside .git/)

  • Objects: Everything (commits, trees, blobs) stored as SHA-1 hashes
  • HEAD: Reference to the current branch
  • Config: Repository configuration
  • Hooks: Scripts executed before/after Git events
  • Info/Exclude: Patterns to exclude from Git tracking (.gitignore)

Git Blob (Binary Large Object)

A blob stores the contents of each file. Git tracks files by computing their SHA-1 hash.

🔍 Inspecting: git cat-file -p <hash> — view blob contents
git cat-file -t <hash> — view blob type (blob, tree, commit)

⚙️ Unit 4 - Advanced Git

4.1 Source Code Management (SCM)

SCM is a set of processes and tools to manage and track changes in source code.

Why SCM Matters

  • Collaboration: Multiple developers work on the same codebase
  • History: Complete record of all changes
  • Reverting: Roll back to stable state when bugs are introduced
  • Branching & Merging: Parallel development streams
  • Conflict Resolution: Identify and resolve conflicting changes

SCM Best Practices

  • Commit often — frequent commits give many revert opportunities
  • Work from the latest version
  • Make detailed commit messages
  • Review changes before committing (use staging area)
  • Use branches for features/fixes
  • Agree on a team workflow (GitFlow, GitHub Flow)

4.3 Rebasing

Rebase moves or combines a sequence of commits to a new base commit. It creates a linear project history (unlike merge which creates merge commits).

Two Modes

ModeCommandDescription
Standardgit rebase <base>Automatically rebases current branch onto base
Interactivegit rebase -i <base>Opens editor to alter individual commits (pick, squash, edit, drop)
BEFORE REBASE (merge): main ●──●──●──────────● \ / feature ●──●──● AFTER REBASE: main ●──●──● \ feature ●──●──●──●──● (linear!)
🎯 Interview Point: Use git merge to preserve complete history; use git rebase for clean, linear history. Never rebase public/shared branches!

📦 Unit 6 - Maven

6.1 What is Maven?

Apache Maven is a build automation and project management tool primarily for Java projects. It manages the build lifecycle, dependencies, documentation, and reporting using a standardized approach.

Key Features

  • Build Automation: Compile, test, package, deploy
  • Dependency Management: Auto-resolves and downloads libraries
  • Standardized Structure: Consistent project layout across teams
  • Plugin System: Extensible through plugins
  • Multi-Module Support: Manage multiple related projects

What is a Maven Artifact?

An artifact is any file used in Java software development — JAR files, POM files, documentation bundles, ML models, etc. The most common artifact is a dependency (Java library).

6.2 Maven Build Lifecycle

Three built-in lifecycles:

1. Default Lifecycle (Build)

validate → compile → test → package → verify → install → deploy
PhaseDescription
validateValidate project configuration is correct
compileCompile source code into bytecode
testRun unit tests
packagePackage into distributable format (JAR/WAR)
verifyRun checks to verify package quality
installInstall to local repository
deployCopy to remote repository

2. Clean Lifecycle

pre-clean → clean → post-clean — Deletes files from previous build (target directory)

3. Site Lifecycle

pre-site → site → post-site → site-deploy — Generates project documentation

📝 Important: Running mvn package automatically runs all phases before it (validate, compile, test, package).

☁️ Cloud Computing

What is Cloud Computing?

Definition (NIST): Cloud computing is a model for enabling ubiquitous, convenient, on-demand network access to a shared pool of configurable computing resources (networks, servers, storage, applications, services) that can be rapidly provisioned and released with minimal management effort.

5 Essential Characteristics (NIST)

  1. On-Demand Self-Service: Users can provision resources automatically without human interaction with the provider
  2. Broad Network Access: Available over the network through standard devices (phones, laptops)
  3. Resource Pooling: Provider's resources are pooled to serve multiple consumers (multi-tenant)
  4. Rapid Elasticity: Resources can scale up/down quickly to match demand
  5. Measured Service: Usage is monitored, controlled, and billed (pay-as-you-go)

🎤 Interview Questions

Top 50 DevOps Interview Questions

1. What is DevOps and why is it important?
DevOps is a set of practices that combines software development (Dev) and IT operations (Ops) to shorten the development lifecycle and provide continuous delivery. It emphasizes automation, monitoring, and collaboration. It's important because it enables faster time-to-market, better quality, and improved reliability.
2. What is the difference between Continuous Delivery and Continuous Deployment?
Continuous Delivery ensures code is always in a deployable state and requires manual approval before production deployment. Continuous Deployment goes further by automatically deploying every change that passes testing to production without human intervention.
3. What is Infrastructure as Code (IaC)?
IaC is managing and provisioning infrastructure through machine-readable configuration files rather than manual processes. Tools like Terraform, Ansible, and CloudFormation enable IaC, ensuring consistent, repeatable, and version-controlled infrastructure deployments.
4. What are the differences between Docker and Virtual Machines?
Docker containers share the host OS kernel (lightweight, MBs, seconds to boot), while VMs include a full guest OS (heavy, GBs, minutes to boot). Containers are more portable and resource-efficient; VMs provide stronger isolation.
5. What is a CI/CD pipeline?
A CI/CD pipeline is an automated workflow that takes code from development to production. It includes stages: Source (version control), Build (compile), Test (automated testing), Staging (pre-production), and Deployment (release to production).
6. What is the difference between Git merge and Git rebase?
Git merge creates a merge commit preserving full history. Git rebase replays commits on top of another base, creating a linear history. Merge is safer for shared branches; rebase is better for cleaning up feature branches before merging.
7. What is the purpose of .gitignore?
.gitignore tells Git which files and directories to ignore/exclude from version control. Common entries include build artifacts, node_modules, .env files, IDE settings, and logs.
8. Explain the Git lifecycle.
1. Working Directory (edit files) → 2. Staging Area (git add) → 3. Local Repository (git commit) → 4. Remote Repository (git push). Reverse: git pull/fetch from remote, git reset from local to working directory.
9. What is container orchestration?
Container orchestration automates the deployment, scaling, management, and networking of containers. Kubernetes is the most popular orchestrator, handling load balancing, auto-scaling, self-healing, and rolling updates.
10. What is the difference between Ansible and Puppet?
Ansible is agentless (uses SSH), simpler, uses YAML playbooks, no agent installation needed. Puppet is agent-based (Master-Slave), uses Ruby DSL manifests, requires agent on each node, better for large-scale environments.

Top 30 Git Interview Questions

1. What is a repository in Git?
A repository (repo) is a storage space that contains all the project files and the complete history of changes. It includes a .git directory with all metadata, objects, and configuration.
2. What is the difference between git pull and git fetch?
git fetch downloads changes from remote but doesn't merge them into your working directory. git pull does both — fetches AND merges remote changes into your current branch.
3. What is a merge conflict and how do you resolve it?
A merge conflict occurs when Git can't automatically merge changes from two branches (same lines modified). Resolution: 1) Open conflicted files, 2) Manually choose which changes to keep, 3) Remove conflict markers, 4) git add and commit.
4. What is git stash used for?
git stash temporarily saves uncommitted changes (both staged and unstaged) and reverts the working directory. Useful when you need to switch branches but aren't ready to commit. Use git stash pop to restore.
5. What is the difference between a local and remote repository?
A local repository exists on your machine with the full project history. A remote repository is hosted on a server (GitHub/GitLab) for team collaboration. Changes move between them via push/pull.
6. What is git cherry-pick?
git cherry-pick allows you to apply a specific commit from one branch to another without merging the entire branch. Useful for applying hotfixes.
7. What is a detached HEAD state?
Detached HEAD occurs when you checkout a specific commit, tag, or remote branch instead of a local branch. Commits made in this state aren't on any branch and can be lost. Use git checkout -b to create a branch.
8. How do you undo the last commit?
git reset --soft HEAD~1 (undo commit, keep changes staged), git reset HEAD~1 (undo commit, keep changes unstaged), git reset --hard HEAD~1 (undo commit and discard all changes). Use --soft for safety.
9. What is git bisect?
git bisect uses binary search to find the commit that introduced a bug. You mark a known good commit and a known bad commit, and Git systematically tests commits in between.
10. What is the difference between git reset, git revert, and git restore?
git reset moves HEAD to a previous commit (rewrites history). git revert creates a new commit that undoes changes (safe for shared branches). git restore discards changes in working directory or staging area.

Top 20 AWS Interview Questions

1. What is EC2 and its pricing models?
EC2 (Elastic Compute Cloud) provides virtual servers in the cloud. Pricing: On-Demand (pay per use), Reserved (1-3 year commitment, 72% savings), Spot (up to 90% off, can be terminated), Savings Plans (flexible commitment).
2. What is the difference between S3 and EBS?
S3 is object storage (files, images, backups) accessible via HTTP. EBS is block storage (like a hard drive) attached to EC2 instances for database and file system use. S3 = unlimited objects; EBS = mounted volumes.
3. What is IAM and its best practices?
IAM (Identity and Access Management) controls who can access what AWS resources. Best practices: use least privilege, enable MFA, avoid root account, use roles not access keys, rotate credentials regularly.
4. What is VPC?
VPC (Virtual Private Cloud) is a logically isolated section of AWS cloud where you can launch resources in a virtual network. You control IP address ranges, subnets, route tables, and network gateways.
5. What is the difference between Security Groups and NACLs?
Security Groups are stateful (return traffic auto-allowed) and operate at instance level. NACLs are stateless (must allow both inbound and outbound) and operate at subnet level. Security Groups are more commonly used.
6. What is an Elastic Load Balancer?
ELB distributes incoming application traffic across multiple targets (EC2, containers, IPs). Types: ALB (Application - HTTP/HTTPS), NLB (Network - TCP/UDP), GLB (Gateway - for on-premise).

Top 20 Linux Interview Questions

1. What is the difference between hard link and soft link?
Hard link points directly to the inode (data). Soft link (symbolic link) is a shortcut pointing to the file path. Hard links survive file deletion; soft links break if original file is deleted.
2. What does chmod 755 mean?
755 = Owner: rwx (7), Group: r-x (5), Others: r-x (5). Owner can read/write/execute. Group and others can read and execute but not write. Common for directories and scripts.
3. What is the difference between kill and kill -9?
kill sends SIGTERM (signal 15) which allows graceful shutdown. kill -9 sends SIGKILL which forcefully terminates the process immediately without cleanup. Use -9 only when graceful termination fails.
4. What is a zombie process?
A zombie process is a process that has completed execution but still has an entry in the process table. It occurs when a child process terminates but the parent hasn't read its exit status. Use ps aux | grep Z to find them.
5. What is the purpose of /etc/passwd and /etc/shadow?
/etc/passwd contains user account info (username, UID, GID, home dir, shell). /etc/shadow contains encrypted passwords and password aging info. Shadow file is more secure with restricted permissions.

5-Mark Questions

Explain the DevOps lifecycle with its phases and tools.
The DevOps lifecycle has 7 phases: 1) Plan (Jira), 2) Code (Git), 3) Build (Maven), 4) Test (Selenium), 5) Release (Jenkins), 6) Deploy (Docker/K8s), 7) Monitor (Nagios). It's a continuous cycle with feedback loops between each phase, enabling rapid and reliable software delivery.
Compare Docker containers and Virtual Machines.
Containers share the host OS kernel (lightweight, MBs, instant boot). VMs have full guest OS (heavy, GBs, minutes to boot). Containers are more portable and efficient; VMs provide stronger isolation. Docker uses containerd; VMs use hypervisors (VMware, Hyper-V). Containers are better for microservices; VMs for running different OS environments.
Explain branching and merging in Git.
Branching creates a separate line of development from the main branch, allowing parallel work without affecting production code. Commands: git branch (create), git checkout (switch). Merging combines branch changes into another (git merge). Merge conflicts occur when the same lines are modified — resolved by manually choosing changes.
What is CI/CD and its benefits?
CI (Continuous Integration) automatically builds and tests code on every commit, catching bugs early. CD (Continuous Delivery/Deployment) automates the release process. Benefits: faster releases, fewer bugs, automated testing, consistent deployments, quick rollback, and better team collaboration.
Explain IaaS, PaaS, and SaaS with examples.
IaaS (Infrastructure as a Service) provides virtual machines and infrastructure — you manage OS, runtime, apps (e.g., AWS EC2). PaaS (Platform as a Service) provides a platform for deploying apps — you manage apps and data only (e.g., Heroku). SaaS (Software as a Service) provides complete applications — you just use them (e.g., Gmail, Salesforce).

10-Mark Questions

Explain the architecture and workflow of Jenkins in a CI/CD pipeline.
Jenkins uses Master-Slave architecture. The Master schedules builds, dispatches to Slaves, monitors, and records results. Slaves are remote Java executables that execute build jobs. Connectivity via SSH (port 22) or JNLP (port 50000). Workflow: Developer commits → Jenkins polls repo → Detects changes → Pulls code → Builds → Runs tests → If passes, deploys to staging → On approval, deploys to production → Feedback to developers. Jenkins supports 1000+ plugins for integration with virtually any DevOps tool. Originally called Hudson, created by Kohsuke Kawaguchi at Sun Microsystems in 2004.
Explain Docker containerization and its role in DevOps.
Docker packages applications with all dependencies into lightweight containers. Architecture layers: Infrastructure → OS → Container Engine → Application + Dependencies. Containers share the host OS kernel (unlike VMs), making them fast and efficient. Docker images are read-only templates; containers are running instances. Role in DevOps: 1) Eliminates "works on my machine" problem, 2) Consistent dev/staging/prod environments, 3) Fast deployment and scaling, 4) Works with CI/CD tools (Jenkins builds Docker images), 5) Orchestrated by Kubernetes for large-scale deployments. AWS services: ECS (container management), Fargate (serverless containers), ECR (container registry).
