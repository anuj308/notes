1. Introduction to Operating System

Definition

An Operating System (OS) is system software that manages all computer resources (hardware and software) and provides an environment for users to execute programs conveniently and efficiently.

Why Does OS Exist?

  • Without OS: Every app would need complex hardware interaction code
  • Without OS: A single app could exploit all resources
  • Without OS: No memory protection between programs

🔧 Real-World Analogy

Think of OS as a restaurant manager — customers (users) place orders (requests), the manager coordinates chefs (CPU), waiters (I/O), and ingredients (memory/resources) to serve food efficiently.

Functions of OS

  • Resource Management: Manages memory, devices, files, security, processes
  • Abstraction: Hides hardware complexity from users
  • Protection & Isolation: Prevents programs from interfering with each other
  • Interface: Provides GUI/CLI for user interaction

Key Goals of OS

  • Maximum CPU utilization
  • Minimum process starvation
  • Higher priority job execution
  • Maximum throughput
🧠 Memory Trick: OS = R.A.P.I — Resource management, Abstraction, Protection, Interface

Single Process OS

Only 1 process executes at a time from the ready queue. Oldest Type

Batch Processing OS

  • User prepares job using punch cards → submits to operator → operator sorts into batches → executes batch by batch
  • ❌ No priority setting, may cause starvation, CPU idle during I/O

Multiprogramming OS

  • Multiple jobs kept in memory; CPU switches when current process goes to wait state
  • Single CPU, context switching
  • ✅ Reduces CPU idle time

Multitasking OS

  • Logical extension of multiprogramming
  • Time sharing + context switching → appears to run multiple tasks simultaneously
  • ✅ Increases responsiveness, further reduces CPU idle time

Multiprocessing OS

  • More than 1 CPU in a single computer
  • ✅ Better reliability (if 1 CPU fails, others work), better throughput, lesser starvation

Distributed OS

  • Manages resources across loosely connected autonomous computer nodes
  • Collection of independent, networked, physically separate computational nodes

Real-Time OS (RTOS)

  • Error-free computations within tight time boundaries
  • Used in: Air Traffic Control, Robots, Medical Devices

Additional Types

  • Network OS (NOS): Supports file sharing, printer sharing in networks
  • Embedded OS: For smartphones, IoT devices, industrial machines
  • Mobile OS: Touch input, app management, wireless communication
  • Multi-User OS: Multiple concurrent users with access control

3. Kernel & Its Types

Definition

The kernel is the core component of the OS that interacts directly with hardware and performs the most crucial tasks. It is the first part of the OS to load on startup.

Functions of Kernel

  • Process Management: Scheduling, creating/deleting processes, synchronization
  • Memory Management: Allocating/deallocating memory, tracking usage
  • File Management: Creating/deleting files and directories
  • I/O Management: Buffering, caching, and spooling

4. System Calls

Definition

A system call is the mechanism by which a user program requests a service from the kernel for which it doesn't have permission. It is the only way a process can go from user mode to kernel mode.

How It Works

User executes command (User Space) → System Call (transition via software interrupt) → Kernel executes service (Kernel Space) → Returns to User Space

Process Control Block (PCB)

Data structure storing: Process ID, Program Counter, Process State, Priority, CPU Registers, Memory limits, I/O status, Accounting info.

Process Types

  • Foreground: Interacts with user, may block UI
  • Background (Daemon): Runs independently, no user interaction
  • Orphan: Parent terminated, still running → adopted by init process
  • Zombie: Execution complete but entry remains in process table until parent reads exit status
🧠 Memory Trick: Orphan = Parent gone, Child alive. Zombie = Child gone, Parent hasn't checked.

Context Switching

Switching CPU to another process — saves current process state (PCB) and loads new process state. Pure overhead (no useful work during switch).

Swapping

Medium-term scheduler removes processes from memory to disk (swap-out) and reintroduces them later (swap-in). Used to reduce degree of multi-programming.

8. Process Synchronization

Critical Section Problem

The segment of code where processes access shared resources. Only one process/thread should be in the critical section at a time.

Desired Properties

  • Mutual Exclusion: Only one process in CS at a time
  • Progress: If no one is in CS, a waiting process must be able to enter
  • Bounded Waiting: Finite bound on how long a process waits

Mutex vs Semaphore

Binary vs Counting Semaphore

  • Binary: Value 0 or 1 (equivalent to mutex)
  • Counting: Range over unrestricted domain (controls finite instances of resources)

Conditional Variables

Used to avoid busy waiting. Thread releases lock, enters wait state until notified. No contention.

Classical Synchronization Problems

  • Producer-Consumer: Producers generate data, consumers consume it
  • Reader-Writer: Multiple readers, exclusive writers
  • Dining Philosophers: 5 philosophers, 5 forks, avoid deadlock
  • Sleeping Barber: Barber and customer synchronization
🧠 Memory Trick: Mutex = Man (one owner). Semaphore = Signal (anyone can signal).

9. Deadlocks

Definition

Deadlock occurs when two or more processes are unable to proceed because each is waiting for the other to release a resource — forever.

Four Necessary Conditions (Coffman)

  1. Mutual Exclusion: Resource is non-sharable
  2. Hold & Wait: Process holds resource while waiting for another
  3. No Preemption: Resources released only voluntarily
  4. Circular Wait: Chain of processes each waiting for the next's resource
🧠 Memory Trick: M.H.N.C — My Horse Never Cries

Methods for Handling Deadlocks

MethodDescriptionApproach
PreventionBreak at least one of the 4 conditionsStatic (design phase)
AvoidanceCheck if allocation leads to safe state before grantingDynamic (runtime)
Detection + RecoveryAllow deadlock, detect it, then recoverRuntime
Ignorance (Ostrich)Pretend deadlocks don't existMost OSes do this!

Deadlock Prevention Details

  • Break Mutual Exclusion: Use sharable resources (read-only files) — can't always do this
  • Break Hold & Wait: Request all resources at once, or release all before requesting new
  • Break No Preemption: Preempt resources from waiting processes (may cause livelock)
  • Break Circular Wait: Impose resource ordering — all processes request in increasing order

Banker's Algorithm (Deadlock Avoidance)

Need[i] = Max[i] - Allocation[i] For each request: 1. Check Request ≤ Need (otherwise error) 2. Check Request ≤ Available (otherwise wait) 3. Pretend to allocate: Available -= Request, Allocation += Request, Need -= Request 4. Run safety algorithm — if safe, grant; else, rollback and wait

Complexity: O(n × m²) where n = processes, m = resource types

Deadlock Detection

  • Single instance per resource type: Wait-for graph — cycle = deadlock
  • Multiple instances: Banker's algorithm (similar approach)

Recovery from Deadlock

  • Process Termination: Abort all deadlocked processes, or one at a time until cycle breaks
  • Resource Preemption: Take resources from processes until cycle breaks

Memory Management Techniques

A. Contiguous Allocation

  • Fixed Partitioning: Memory divided into fixed-size partitions → Internal fragmentation, external fragmentation, limited process size
  • Dynamic Partitioning: Partition size decided at load time → No internal fragmentation, but external fragmentation exists

B. Paging (Non-Contiguous)

Physical memory → fixed-size Frames. Logical memory → same-size Pages. Eliminates external fragmentation.

  • Page Table: Maps page number → frame number. Stored in main memory, base address in PCB.
  • PTBR: Page Table Base Register — points to current page table. Single register update on context switch.
  • TLB (Translation Lookaside Buffer): Hardware cache for page table entries. Speeds up address translation. ASID identifies each process.

C. Segmentation (Non-Contiguous)

Divides process into variable-sized segments based on user view (code, data, stack, heap). Address: <segment-number, offset>

  • ✅ No internal fragmentation, user-friendly view
  • ❌ External fragmentation, complex swapping

Virtual Memory

Allows execution of processes not completely in memory. Programs can be larger than physical RAM.

Demand Paging

  • Pages loaded only when needed (lazy swapper/pager)
  • Valid-Invalid Bit: 1 = page in memory; 0 = page on disk or invalid
  • Page Fault: Accessing invalid page → trap to OS → find free frame → load page from disk → restart instruction

Page Replacement Algorithms

Thrashing

When a process spends more time paging (servicing page faults) than executing. Occurs when total demand for frames exceeds available frames.

Thrashing Solutions:
  • Working Set Model: Allocate frames based on current locality. If frames < locality size → thrashing.
  • Page Fault Frequency: Set upper/lower bounds on page fault rate. Too high → add frames. Too low → remove frames.

🎤 Top Placement Interview Questions 30+ Key Questions

1. What is an Operating System?

Short Answer (30s): An OS is system software that manages hardware resources, provides services to applications, and acts as an interface between users and computer hardware.
Detailed Answer (2-3 min): An Operating System is the most fundamental software layer that sits between computer hardware and user applications. It performs four key functions: Resource Management (CPU scheduling, memory allocation, I/O device management), Abstraction (hides hardware complexity), Protection (isolates processes from each other), and User Interface (GUI/CLI). Without an OS, every application would need to directly manage hardware, leading to conflicts and inefficiency. Modern examples include Linux, Windows, macOS.
Follow-up: Can you explain the difference between a kernel and an OS?

2. Difference between Process and Thread?

Short Answer: A process is an executing program with its own memory space. A thread is a lightweight unit of execution within a process that shares memory with other threads.
Detailed Answer: A Process has its own address space, PCB, resources, and memory protection. Creating a process is expensive. A Thread shares address space, files, and resources with other threads in the same process but has its own program counter, registers, and stack. Thread context switching is faster (no memory space switch). Benefits of multithreading: responsiveness, resource sharing, economy, and better CPU utilization on multi-core systems.
Follow-up: When would you use threads over processes?

3. What is Deadlock? Explain Prevention vs Avoidance.

Short Answer: Deadlock is when processes are stuck waiting for each other's resources forever. Prevention breaks one of the 4 necessary conditions. Avoidance checks safety before allocating.
Detailed Answer: Deadlock occurs when 4 conditions hold simultaneously: Mutual Exclusion, Hold & Wait, No Preemption, Circular Wait. Prevention is a static approach — we break one condition at design time (e.g., request all resources at once to break Hold & Wait). Avoidance is dynamic — we check if granting a request keeps the system in a safe state using Banker's Algorithm. The trade-off: Prevention is restrictive (lower resource utilization) but simpler. Avoidance is flexible but adds runtime overhead and may still underutilize resources.
Follow-up: What is a safe sequence? Can you walk through Banker's Algorithm?

4. Paging vs Segmentation?

Short Answer: Paging divides memory into fixed-size blocks (OS view). Segmentation divides into variable-size logical blocks (user view).
Detailed Answer: Paging divides physical memory into fixed-size frames and logical memory into same-size pages. It eliminates external fragmentation but may cause internal fragmentation. Address: page number + offset. Uses page table for mapping. Segmentation divides process into variable-sized segments based on logical units (code, data, stack). It supports user's view of memory but causes external fragmentation. Address: segment number + offset. Uses segment table with base and limit. Modern systems often use a combination of both.
Follow-up: What is TLB and why is it needed?

5. What is Virtual Memory?

Short Answer: Virtual memory gives programs the illusion of large contiguous memory by using disk as extension of RAM. Pages are loaded on demand.
Detailed Answer: Virtual memory allows execution of processes not fully loaded in physical memory. The OS treats a portion of secondary storage (swap space) as main memory. Key mechanism: Demand Paging — only load pages when needed. When a process accesses a page not in memory → Page Fault → OS loads it from disk → may need to replace an existing page. Advantages: programs larger than RAM can run, more processes in memory (higher multiprogramming). Disadvantages: overhead of page replacement, potential thrashing.
Follow-up: What is thrashing and how do you handle it?

6. Semaphore vs Mutex?

Short Answer: Mutex is binary (locked/unlocked), same thread locks/unlocks. Semaphore is an integer counter, any thread can signal.
Detailed Answer: Mutex provides exclusive access — only the thread that locks it can unlock it. Simple, used for mutual exclusion. Semaphore is more versatile — it's an integer where wait() decrements and signal() increments. Binary semaphore (0/1) is similar to mutex. Counting semaphore can control access to a pool of resources. Any thread can call signal() on a semaphore, unlike mutex. Semaphores are used for signaling and coordination, not just mutual exclusion.
Follow-up: How would you solve the Producer-Consumer problem?

7. Explain Round Robin Scheduling.

Short Answer: Each process gets a fixed time quantum. When time expires, process goes to back of queue. Fair, preemptive, designed for time-sharing.
Detailed Answer: Round Robin is the most popular preemptive algorithm. Each process gets a time quantum (TQ). CPU switches to next process when TQ expires. If TQ is too small: excessive context switching overhead. If TQ is too large: degenerates to FCFS. No starvation, no convoy effect. Average waiting time depends on TQ and number of processes. Performance degrades with more processes as each gets smaller time slice.
Follow-up: How do you determine optimal TQ?

8. What is a System Call? Give examples.

Short Answer: System calls are the interface between user applications and the OS kernel. They allow user programs to request privileged operations.
Detailed Answer: When a user program needs hardware access or privileged operations (file I/O, process creation, memory allocation), it makes a system call. This triggers a software interrupt, switching from user mode to kernel mode. Types include: Process Control (fork, exit, wait), File Management (open, read, write), Device Management (ioctl), Communication (pipe, shmget), and Information (getpid, time). After the kernel completes the service, control returns to user mode.
Follow-up: What happens during a mode switch from user to kernel?

9. What are Zombie and Orphan Processes?

Short Answer: Zombie = completed child process waiting for parent to read exit status. Orphan = child whose parent has terminated, adopted by init.
Detailed Answer: A Zombie process has finished execution but still has an entry in the process table because the parent hasn't called wait() to read its exit status. Zombies consume process table entries but no other resources. They're "reaped" when the parent calls wait(). An Orphan process is a running child whose parent has terminated. Orphans are automatically adopted by the init process (PID 1), which eventually reaps them. Zombies are a concern for resource leaks; orphans are generally not problematic.

10. What is Thrashing? How to handle it?

Short Answer: Thrashing = system spends more time paging than executing. Happens when processes don't have enough frames for their working set.
Detailed Answer: Thrashing occurs when total demand for frames exceeds physical memory, causing excessive page faults and swapping. Solutions: 1. Working Set Model: Track each process's working set (set of pages in active use). Allocate enough frames for the working set. 2. Page Fault Frequency (PFF): Set upper and lower bounds on page fault rate. If rate > upper bound → allocate more frames. If rate < lower bound → reclaim frames. 3. Swapping out entire processes to reduce multiprogramming level.

❓ Frequently Asked Questions

Q: What is the difference between fork() and exec()?

Answer: fork() creates a child process by duplicating the parent process. Both parent and child run the same code. exec() replaces the current process's memory image with a new program. Typically, fork() is called first to create a child, then exec() is called in the child to load a different program.

Related: wait(), exit(), zombie process, process creation

Q: Why is context switching considered overhead?

Answer: During context switching, the CPU saves the state of the current process (registers, PC, memory maps) and loads the state of the next process. No useful computation is done during this time. It also flushes the CPU cache, which can cause cache misses when the new process runs. Speed depends on memory speed and number of registers.

Q: What is Internal vs External Fragmentation?

Answer: Internal Fragmentation: Space inside an allocated block is wasted because the process is smaller than the block. E.g., fixed partitioning where a 50KB process uses a 100KB partition. External Fragmentation: Total free memory is enough, but it's not contiguous. E.g., after dynamic partitioning, scattered free holes can't fit a large process. Paging eliminates external fragmentation but may have internal fragmentation (last page).

Q: What is the complexity of Banker's Algorithm?

Answer: O(n × m²) where n = number of processes and m = number of resource types. The safety algorithm runs in O(m × n²). For each process, we check if Need ≤ Work (O(m)), and there are n processes. This runs potentially n times. In practice, it's run when processes request resources, not continuously.

Q: What is Pure Demand Paging?

Answer: In pure demand paging, a process starts with NO pages in memory. Every instruction access causes a page fault, and pages are loaded only when demanded. This is the extreme case of demand paging. In practice, we use locality of reference — a process tends to access nearby pages — to achieve reasonable performance.

Q: Why do we need User Mode and Kernel Mode?

Answer: Protection and security. User mode restricts access to privileged instructions (direct hardware access, memory management). Kernel mode has full access. This prevents buggy or malicious programs from crashing the system or accessing other processes' memory. Transitions happen via system calls (user→kernel) and interrupts/traps (kernel→user).

🔥 Must-Know Definitions

  • OS: Resource manager + abstraction layer between hardware and applications
  • Process: Program under execution with own memory space
  • Thread: Lightweight unit of execution sharing process memory
  • System Call: Interface between user programs and kernel (user mode → kernel mode)
  • Deadlock: Processes stuck waiting for each other (4 conditions: MHNC)
  • Paging: Non-contiguous memory allocation using fixed-size pages and frames
  • Virtual Memory: Illusion of large memory using disk as extension of RAM
  • Semaphore: Integer-based synchronization primitive (binary: 0/1, counting: 0..n)
  • Mutex: Binary lock for mutual exclusion, same thread locks/unlocks
  • Thrashing: Excessive paging activity; system spends more time paging than executing
  • Zombie: Completed process still in process table (parent hasn't reaped)
  • Orphan: Running child whose parent terminated (adopted by init)

🧠 Memory Tricks

Deadlock Conditions: M.H.N.C — My Horse Never Cries (Mutual exclusion, Hold & Wait, No preemption, Circular wait)
Kernel Types: Mono = Everything, Micro = Minimal, Hybrid = Both
OS Functions: R.A.P.I — Resource management, Abstraction, Protection, Interface
Page Replacement: FIFO = oldest, LRU = least recent, Optimal = future (impossible)
Fragmentation: Internal = wasted space INSIDE allocated block, External = not enough CONTIGUOUS free space

⚠️ Common Exam Mistakes

  • Confusing orphan (parent gone) with zombie (child done, parent hasn't checked)
  • Thinking multithreading helps on single CPU — it doesn't (just context switching overhead)
  • Forgetting that FIFO can have Belady's Anomaly but LRU/Optimal cannot
  • Confusing internal (within partition) vs external (between partitions) fragmentation
  • Believing deadlock prevention and deadlock avoidance are the same — prevention is static, avoidance is dynamic
  • Thinking mutex and binary semaphore are identical — mutex has owner concept

🚀 30-Minute Placement Crash Course

🎯 Top 10 Interview Questions (Quick Answers)

  1. Process vs Thread? Process = own memory. Thread = shares memory, own stack/PC.
  2. Deadlock conditions? MHNC: Mutual exclusion, Hold & Wait, No preemption, Circular wait.
  3. Paging vs Segmentation? Paging = fixed blocks (OS view). Segmentation = variable logical blocks (user view).
  4. Mutex vs Semaphore? Mutex = binary, owner-locked. Semaphore = integer counter, any thread signals.
  5. Virtual Memory? Using disk as RAM extension. Pages loaded on demand via page faults.
  6. Banker's Algorithm? Deadlock avoidance. Checks if allocation keeps system in safe state before granting.
  7. Round Robin? Preemptive, fixed TQ. Fair, no starvation. Small TQ = overhead, large TQ = FCFS.
  8. Zombie vs Orphan? Zombie = done but parent hasn't reaped. Orphan = parent terminated, adopted by init.
  9. TLB? Hardware cache for page table. TLB hit = fast, TLB miss = access page table.
  10. Thrashing? Excessive page faulting. Fix: Working Set Model or PFF.

🔥 Most Asked MCQ Patterns

  • Deadlock condition identification → Know the 4 conditions (MHNC)
  • Scheduling algorithm comparison → FCFS (convoy), SJF (best avg WT), RR (fair), Priority (starvation)
  • Page replacement → FIFO (Belady's), LRU (no Belady's), Optimal (impossible)
  • Memory allocation → Fixed (internal frag), Dynamic (external frag), Paging (no external frag)
  • Kernel types → Monolithic (Linux), Micro (MINIX), Hybrid (Windows, macOS)

CPU Scheduling Formulas

Turnaround Time (TAT) = Completion Time - Arrival Time Waiting Time (WT) = Turnaround Time - Burst Time Response Time = First CPU Allocation Time - Arrival Time Throughput = Number of Processes Completed / Total Time CPU Utilization = (CPU Busy Time / Total Time) × 100% Average WT = Σ(Waiting Time of each process) / Number of processes Average TAT = Σ(TAT of each process) / Number of processes

Page Table Size = Logical Address Space / Page Size Internal Fragmentation = Partition Size - Process Size External Fragmentation = Total Free Memory - Largest Free Block Page Fault Rate = Number of Page Faults / Number of Memory Accesses EAT = (1 - p) × Memory Access Time + p × Page Fault Service Time Disk Access Time = Seek Time + Rotational Latency + Transfer Time Effective Access Time = Hit Ratio × Cache Time + Miss Ratio × Main Memory Time

Disk Space Efficiency = (Allocated File Space / Total Disk Space) × 100% Records per Block = Block Size / Record Size FAT Size = Total Disk Space / (Block Size × FAT Entry Size) Disk I/O Time = Seek Time + Rotational Delay + Transfer Time + Overhead Compression Ratio = Original File Size / Compressed File Size
