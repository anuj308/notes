1. Introduction to Operating System ▶

Definition: An Operating System (OS) is system software that acts as an intermediary between computer hardware and user programs. It manages hardware resources, provides a stable environment for applications, and offers a user interface.

Why it exists: Without an OS, every program would need to know every hardware detail. The OS abstracts complexity, manages resources efficiently, and enables multitasking.

Real-world analogy: The OS is like a hotel manager. Guests (applications) request rooms (memory), food (CPU time), cleaning (I/O) — the manager coordinates everything so guests don't interfere with each other.

Goals of an OS:

  • Convenience: Makes computer easier to use
  • Efficiency: Manages resources optimally
  • Extensibility: Allows development and testing of new features
FunctionDescription
Process ManagementCreate, schedule, and terminate processes
Memory ManagementAllocate and deallocate memory to processes
File ManagementOrganize files and directories on storage
Device ManagementManage I/O devices via device drivers
Security & ProtectionControl access to resources
User InterfaceProvide CLI or GUI for user interaction
Exam Tip: Be able to list the 5 major services of an OS: Process mgmt, Memory mgmt, File mgmt, I/O mgmt, Security.

Batch Operating System

Jobs with similar requirements are grouped into batches and executed without user interaction.

AspectDetails
ProsHigh throughput for repetitive jobs; minimal user intervention
ConsNo user interaction during execution; difficult to debug
ExampleIBM's OS/360, payroll processing systems

Multiprogramming OS

Multiple programs are loaded in memory simultaneously. When one program waits for I/O, the CPU switches to another. Goal: maximize CPU utilization.

Key Concept: CPU is never idle — while one process waits for I/O, another executes.

Multitasking OS

Extension of multiprogramming. CPU rapidly switches between processes, giving users the illusion that multiple programs run simultaneously. Also called Time-Sharing System.

Example: Windows 10/11, macOS, Linux desktop

Time-Sharing OS

Each task gets a fixed time slice (quantum). CPU switches between tasks so fast that users feel they have dedicated machines.

Advantage: Reduces response time, good for interactive use.

Distributed OS

Multiple independent computers appear as a single system to the user. Resources (memory, CPU, storage) are shared across a network.

Examples: Google Search infrastructure, cloud computing platforms

Real-Time OS (RTOS)

Guarantees that critical tasks complete within a specific time constraint. Used in embedded systems, medical devices, avionics.

Key difference from general OS: Deterministic timing — response time is predictable and bounded.

Types: Hard RTOS (deadline must be met — e.g., airbag system) vs. Soft RTOS (deadline can occasionally be missed — e.g., video streaming).

Network OS

Runs on a server and provides network capabilities — file sharing, printer access, security, and communication between connected computers.

Examples: Windows Server, Linux Server distributions

3. System Calls ▶

Definition: System calls are the programming interface through which user programs request services from the OS kernel.

Why it exists: User programs cannot directly access hardware due to security restrictions. System calls provide a controlled gateway.

Real-world analogy: System calls are like ordering from a restaurant menu. You tell the waiter (system call), the chef (kernel) prepares it, and the waiter brings it back.

6. Process Management ▶

Definition: A process is a program in execution — an active entity with its own memory space, registers, and execution context. While a program is a passive set of instructions stored on disk, a process is the actual running instance.

Real-world analogy: A program is like a recipe book. A process is the act of cooking that recipe — you have ingredients (data), tools (registers), and active steps (instructions).

Process Control Block (PCB)

Each process is represented by a PCB (also called Task Control Block). It contains:

FieldDescription
Process ID (PID)Unique identifier
Program CounterAddress of next instruction
CPU RegistersRegister values when process is not running
Memory LimitsBase and limit registers
Open File ListFiles currently opened by the process
Process StateCurrent state (New, Ready, Running, etc.)
PriorityScheduling priority

Process States

5-State Process Model: New → Ready → Running → Waiting → Terminated

Special Process Types

TypeDescription
Orphan ProcessA process whose parent has terminated. On Unix/Linux, orphan processes are adopted by init (PID 1).
Zombie ProcessA process that has completed but still has an entry in the process table (PCB retained). The parent needs to call wait() to reap it. If parent never calls wait(), the zombie remains.
Daemon ProcessA background process that runs continuously, typically started at boot time (e.g., sshd, crond, httpd).
Foreground ProcessProcess with user interaction — occupies the terminal.
Background ProcessProcess running without terminal interaction.
Common Mistake: Confusing "program" with "process". A program is passive (on disk); a process is active (in memory). One program can create multiple processes.
7. Threads ▶

Definition: A thread is the smallest unit of execution within a process. A process can have multiple threads that share the same memory space and resources but execute independently.

Why threads exist: Creating a new process is expensive (PCB, memory allocation). Threads are lightweight — they share resources of their parent process, making context switching cheaper.
User-Level
Kernel-Level

User-Level Threads (ULT)

Managed entirely in user space — the kernel has no knowledge of threads. Thread management is done by a thread library (e.g., POSIX Pthreads).

Pros: Fast context switching, no kernel involvement, works on any OS

Cons: If one thread blocks, the entire process blocks; cannot leverage multiple cores

Kernel-Level Threads (KLT)

Managed directly by the OS kernel. Each thread is individually scheduled.

Pros: Can run on multiple cores simultaneously; if one thread blocks, others continue

Cons: Slower context switching (requires kernel mode switch); more overhead

HybridMany ULTs → Many KLTs (with binding)Combines many-to-one and one-to-oneVery complexModern Solaris
8. CPU Scheduling ▶

Definition: CPU scheduling is the mechanism by which the OS decides which process in the ready queue gets the CPU for execution. The component that makes this decision is called the Short-Term Scheduler (or CPU Scheduler).

Scheduling Metrics

Turnaround Time = Completion Time - Arrival Time
Waiting Time = Turnaround Time - Burst Time
Response Time = First Execution Time - Arrival Time
Throughput = Number of processes completed per unit time
CPU Utilization = (CPU Busy Time / Total Time) × 100%

Scheduling Algorithms

FCFS
SJF
Priority
Round Robin
MLQ/MLFQ
HRRN

First-Come, First-Served (FCFS)

Processes are executed in the order they arrive. Non-preemptive.

ProsCons
Simple, fair in orderingSuffers from Convoy Effect — short processes wait behind long ones
Easy to implement with FIFO queueAverage waiting time can be very high
Convoy Effect: A long CPU-bound process ahead of many short I/O-bound processes causes all I/O devices to remain idle, wasting resources.

Shortest Job First (SJF) / Shortest Job Next (SJN)

Executes the process with the smallest burst time first. Can be preemptive (SRTF — Shortest Remaining Time First) or non-preemptive.

ProsCons
Provably optimal — minimizes average waiting timeStarvation — long processes may never get CPU
Highest throughput for batch systemsRequires knowing burst time in advance (impractical)

Priority Scheduling

Each process has a priority; the highest priority process runs first. Can be preemptive or non-preemptive.

Problem: Starvation — low-priority processes may never run. Solution: Aging — gradually increase priority of waiting processes.

Round Robin (RR)

Each process gets a fixed time slice (quantum). The CPU cycles through processes in a circular queue. Preemptive by nature.

Quantum SizeEffect
Too largeDegrades to FCFS — poor response time
Too smallToo many context switches — low CPU utilization
Ideal (10-100ms)Balances response time and context switch overhead
Rule of thumb: Quantum should be slightly larger than 80% of CPU bursts for optimal performance.

Multilevel Queue (MLQ) & Multilevel Feedback Queue (MLFQ)

MLQ: Processes are divided into separate queues based on type (foreground, background, system). Each queue has its own scheduling algorithm. No movement between queues.

MLFQ: Processes can move between queues. A process that uses too much CPU gets demoted; an I/O-bound process gets promoted. Used in Unix, Windows, macOS.

Highest Response Ratio Next (HRRN)

Non-preemptive. Selects the process with the highest response ratio:

Response Ratio = (Waiting Time + Burst Time) / Burst Time

Favors short jobs but prevents starvation — a long job's ratio increases the longer it waits.

9. Process Synchronization ▶

Definition: Process synchronization ensures that concurrent processes cooperate safely when accessing shared resources, preventing race conditions and data inconsistencies.

Critical Section Problem

A critical section is code that accesses shared resources. Only one process should execute its critical section at a time. Requirements:

RequirementDescription
1. Mutual ExclusionOnly one process at a time can execute its critical section
2. ProgressIf no process is in the critical section, the decision to enter cannot be postponed indefinitely
3. Bounded WaitingA process that wants to enter will eventually get a turn (no starvation)

Synchronization Tools

ToolDescriptionKey Feature
Mutex (Mutual Exclusion)A lock that only one thread can acquire at a time. Like a single-key bathroom lock.Binary — locked or unlocked. Same thread must lock and unlock.
SemaphoreA counter-based synchronization primitive. Can be used for both mutual exclusion and signaling.Counting — can allow N threads. wait() and signal() can be called by different threads.
MonitorHigh-level synchronization construct with condition variables. Only one thread can execute inside the monitor at a time.Cleaner than semaphores; less error-prone.
Condition VariableUsed with mutex to wait for a specific condition to become true.wait(mutex) and signal() / broadcast()
Mutex vs Semaphore (quick): Mutex is a lock (same thread acquires and releases). Semaphore is a signal (one thread signals, another waits). A binary semaphore can be used as a mutex, but they are conceptually different.

Classic Synchronization Problems

ProblemDescriptionKey Challenge
Producer-ConsumerProducer adds data to a shared buffer; consumer removes it. Buffer has finite size.Producer must not add to full buffer; consumer must not remove from empty buffer.
Reader-WriterMultiple readers can read simultaneously; writers need exclusive access.Writer starvation — if readers keep arriving, writer may never get access.
Dining Philosophers5 philosophers sit at a round table with 5 chopsticks (one between each). Each needs 2 chopsticks to eat.Deadlock if everyone picks up left chopstick simultaneously. Starvation possible.
Sleeping BarberOne barber, N chairs. If no customers, barber sleeps. Customer wakes barber.Coordination without busy waiting.
10. Deadlocks ▶

Definition: A deadlock is a situation where two or more processes are unable to proceed because each is waiting for resources held by the others.

Real-world analogy: Two cars meet at a 4-way intersection. Each car waits for the other to go first. Neither moves — deadlock!

Necessary Conditions (Coffman Conditions)

All four must hold simultaneously for a deadlock to occur:

#ConditionDescription
1Mutual ExclusionAt least one resource must be held in non-sharable mode
2Hold and WaitA process holding at least one resource is waiting for additional resources held by others
3No PreemptionResources cannot be forcibly taken from a process
4Circular WaitThere exists a circular chain of processes, each waiting for a resource held by the next

Deadlock Management Strategies

StrategyApproachExample
PreventionEnsure at least one Coffman condition never holdsMake all resources sharable (violates condition 1); Require all resources upfront (violates condition 2); Preempt resources (violates condition 3); Order resources (violates condition 4)
AvoidanceOS knows future resource needs and only grants safe requestsBanker's Algorithm — only allocate if resulting state is safe
DetectionAllow deadlocks, then detect themWait-for graph — if cycle exists, deadlock detected
RecoveryKill processes or preempt resourcesAbort all deadlocked processes or abort one at a time

Banker's Algorithm

A deadlock avoidance algorithm that simulates resource allocation to check if the resulting state is safe. A state is safe if there exists a sequence of processes that can all complete.

Data structures: Available[m], Max[n][m], Allocation[n][m], Need[n][m] where Need = Max - Allocation.
11. Memory Management ▶

Definition: Memory management handles how the OS allocates and deallocates memory to processes, translates logical addresses to physical addresses, and manages the memory hierarchy.

Address Binding

StageTypeWhen binding occurs
Compile TimeAbsolute codeMust know absolute memory location at compile time
Load TimeRelocatable codeBinding when program is loaded into memory
Execution TimeDynamic bindingBinding at runtime (requires hardware support — MMU)

Contiguous Allocation

Each process occupies a single contiguous block of memory.

TypeDescriptionProblem
Fixed PartitioningMemory divided into fixed-size partitionsInternal fragmentation
Dynamic PartitioningPartitions created on demand based on process sizeExternal fragmentation

Free space management algorithms: First Fit (fastest), Best Fit (minimizes waste), Worst Fit (leaves large holes), Next Fit (variant of First Fit).

Non-Contiguous Allocation

Process memory can be scattered across physical memory. Two major approaches:

Paging
Segmentation
Seg + Paging

Paging

Logical memory is divided into fixed-size pages; physical memory into same-sized frames. A page table maps pages to frames.

Advantages: No external fragmentation; efficient use of memory

Disadvantage: Internal fragmentation (last page may be partially used); page table overhead

Page Number (p) = Logical Address / Page Size
Offset (d) = Logical Address % Page Size
Physical Address = Frame Number × Frame Size + Offset

Segmentation

Memory is divided into variable-sized segments based on logical divisions (code, data, stack, heap). A segment table maps segment number to base address and limit.

Advantage: Matches user's view of memory; no internal fragmentation

Disadvantage: External fragmentation; complex allocation

Segmentation with Paging

Combines both: segments are divided into pages. Provides the logical view of segmentation with the memory efficiency of paging. Used in modern OS like Intel x86.

Translation Lookaside Buffer (TLB)

A hardware cache that stores recent page table entries to speed up address translation. Saves one memory access per lookup when there's a TLB hit.

Effective Access Time (EAT): If TLB hit ratio = α, TLB access time = ε, memory access time = t, then EAT = α(ε + t) + (1 - α)(ε + 2t).
12. Virtual Memory ▶

Definition: Virtual memory is a memory management technique that gives a process the illusion of having a large, contiguous address space by using secondary storage (disk) as an extension of RAM.

Why it exists: Physical RAM is limited and expensive. Virtual memory allows programs larger than physical RAM to run, and allows multiple programs to share memory safely.

Demand Paging

Only load pages into memory when they are actually needed (on demand). Managed by the pager or lazy swapper.

Page Fault: When a process accesses a page not in memory. The OS traps the fault, finds a free frame, loads the page from disk, updates the page table, and retries the instruction.

Page Replacement Algorithms

When memory is full and a page fault occurs, the OS must decide which page to evict.

AlgorithmStrategyProsCons
FIFOReplace oldest pageSimpleBelady's Anomaly — more frames can increase page faults
Optimal (OPT)Replace page that won't be used for the longest timeMinimum page faultsImpossible — needs future knowledge (theoretical benchmark)
LRUReplace page not used for the longest timeGood performance; no Belady's AnomalyRequires hardware support (counters or stack)
LFUReplace page with fewest referencesSensible for certain workloadsOld frequently-used pages stay forever
Clock (Second Chance)Circular queue with reference bit; give pages a second chanceEfficient approximation of LRULess accurate than true LRU

Thrashing

Definition: A condition where the system spends more time swapping pages in and out than executing processes. CPU utilization drops sharply.

Causes: Too many processes in memory → each process has too few frames → high page fault rate → processes wait for I/O → CPU idle → OS loads more processes → cycle worsens.

Solutions:

  • Working Set Model: Ensure each process has enough frames for its current locality
  • Page Fault Frequency (PFF): Adjust number of frames allocated based on page fault rate

14. Disk Scheduling ▶

Definition: Disk scheduling algorithms decide the order in which disk I/O requests are serviced to minimize seek time (the time to move the disk arm to the required track).

AlgorithmStrategyProsCons
FCFSServe requests in arrival orderFair, simpleHigh seek time (no optimization)
SSTFServe the request closest to current head positionBetter throughput than FCFSStarvation for far-away requests
SCAN (Elevator)Arm moves in one direction, serving all requests until end, then reversesNo starvation; fairRequests just visited must wait for full cycle
C-SCANSCAN but only serves in one direction; after end, jumps back to startMore uniform wait timeMore overhead from the jump
LOOKSCAN but only goes as far as the last request in each directionMore efficient than pure SCANSimilar to SCAN
C-LOOKLOOK but only serves in one direction, then jumps backMost commonly used in modern systems-
Disk structure: Platters → Tracks → Sectors → Cylinders. Seek time is the most significant component (40-60% of total access time).
15. I/O Management ▶

The OS manages I/O through a layered approach: User-level I/O library → Device-independent OS layer → Device drivers → Hardware controllers → Devices.

I/O Techniques

TechniqueDescriptionCPU Involvement
Programmed I/OCPU waits in a loop checking device status (busy-waiting)High — wastes CPU cycles
Interrupt-Driven I/ODevice sends interrupt when ready; CPU can do other workModerate — interrupt overhead
DMA (Direct Memory Access)DMA controller transfers data directly between device and memory without CPU involvementMinimal — only setup and interrupt
Key Insight: DMA is critical for high-speed I/O (disks, graphics, networks). Without DMA, the CPU would be constantly interrupted, severely limiting throughput.

Buffering & Spooling

Buffering: Storing data in a buffer while in transit between devices. Smooths out speed differences.
Spooling (Simultaneous Peripheral Operations Online): Using disk as a large buffer for devices like printers. Multiple processes can write to the spool without waiting.

16. Security & Protection ▶

Protection is an internal mechanism that controls access to system resources. Security is the external defense against threats.

  • Trojan Horse: Malicious program disguised as legitimate
  • Buffer Overflow: Writing beyond allocated memory bounds
  • Privilege Escalation: Gaining unauthorized higher access rights

Q6. Explain the concept of context switching. ▶
📋 Short Answer (30 sec)

Context switching is the process of saving the state (registers, PC, stack pointer) of the current running process and loading the saved state of another process so it can resume execution.

Q8. Explain the concept of fork(), exec(), and wait() system calls. ▶
📋 Short Answer (30 sec)

Q11. What is thrashing? How do you handle it? ▶

  • Page Fault Frequency (PFF): If a process's page fault rate is too high, give it more frames. If too low, take away frames
  • Reduce degree of multiprogramming: Swap out some processes entirely

Q14. What is the Producer-Consumer problem? How do you solve it? ▶
📋 Short Answer (30 sec)

The Producer-Consumer problem involves a producer adding data to a bounded buffer and a consumer removing data. Both must coordinate to avoid buffer overflow (producer) or underflow (consumer). Solved with semaphores or monitors.

📝 Detailed Answer (2-3 min)

We use three semaphores:

  • mutex = 1: Ensures mutual exclusion for buffer access
  • empty = N: Counts empty slots (producer waits if 0)
  • full = 0: Counts filled slots (consumer waits if 0)
while (true) { // Producer
  produce_item();
  wait(empty); // Wait for empty slot
  wait(mutex); // Enter critical section
  add_to_buffer();
  signal(mutex); // Leave critical section
  signal(full); // Signal consumer
}

while (true) { // Consumer
  wait(full); // Wait for filled slot
  wait(mutex); // Enter critical section
  remove_item();
  signal(mutex); // Leave critical section
  signal(empty); // Signal producer
  consume_item();
}

More Quick Interview Questions

Q18. What is a system call and how does it work? ▶
📋 Answer

A system call is how a user program requests a service from the OS kernel (like reading a file or creating a process). The program traps into kernel mode (via a software interrupt), the kernel performs the operation, and returns the result. Common examples: fopen(), fork(), malloc().

Q19. What is the difference between internal and external fragmentation? ▶
📋 Answer

Internal fragmentation: Wasted space within a fixed-size block (e.g., a process needs 17KB but gets a 20KB partition — 3KB wasted). Common in paging and fixed partitioning.

External fragmentation: Free memory exists but is scattered in small holes — no single block is large enough for a process (e.g., 50MB free in total but all in 5MB chunks — can't run a 20MB process). Common in segmentation and dynamic partitioning.

Q20. Explain Belady's Anomaly. ▶
📋 Answer

Belady's Anomaly is a counterintuitive phenomenon where increasing the number of page frames can increase the number of page faults. It occurs in FIFO page replacement. Other algorithms like LRU and Optimal don't suffer from this anomaly (they follow the stack property).

Q21. What is the difference between a mutex and a semaphore? ▶
📋 Answer

Mutex: Binary (0/1). Only the thread that acquired it can release it. Used for mutual exclusion. Like a single-key bathroom lock.

Semaphore: Counting (N≥0). Any thread can signal (post/increment) or wait (decrement). Used for both mutual exclusion and signaling. Like a parking lot counter.

A binary semaphore can be used like a mutex, but they're conceptually different — mutex has ownership, semaphore does not.

Q22. What is the difference between paging and segmentation? ▶
📋 Answer

Paging: Fixed-size blocks (pages). No external fragmentation. User's view is linear. Page table maps page→frame. Internal fragmentation possible.

Segmentation: Variable-size blocks based on logical divisions (code, data, stack). External fragmentation. Matches user's view. Segment table maps segment→base+limit.

Modern systems often use segmentation with paging for the best of both worlds.

Q23. What is the difference between monolithic kernel and microkernel? ▶
📋 Answer

Monolithic: Entire OS runs in kernel space. Fast but a bug can crash everything. Example: Linux.

Microkernel: Minimal kernel (IPC, basic scheduling). Services run in user space. Stable/secure but slower due to message passing. Example: Minix, QNX.

Hybrid kernels (like Windows NT, macOS XNU) combine aspects of both.

Q24. What is the difference between hard and soft real-time systems? ▶
📋 Answer

Hard RTOS: Missing a deadline is catastrophic (airbag system, pacemaker, flight control).

Soft RTOS: Missing a deadline degrades quality but is tolerable (video streaming, online gaming, voice calls).

Q25. What is the Banker's Algorithm? ▶
📋 Answer

The Banker's Algorithm is a deadlock avoidance algorithm that simulates resource allocation to determine if the resulting state is safe. Data structures: Available (resources free), Max (maximum demand of each process), Allocation (currently held), Need (Max - Allocation).

When a process requests resources, the OS pretends to grant them and checks if there exists a safe sequence where all processes can complete. If yes, the request is granted; otherwise, the process waits.

Q26. What is the purpose of the TLB? ▶
📋 Answer

The Translation Lookaside Buffer (TLB) is a hardware cache that stores recent virtual-to-physical address translations. It speeds up paging by avoiding a full page table lookup in memory. On a TLB hit, translation takes one cycle. On a miss, the system must access the page table (slower).

Q27. Explain the read-write lock concept. ▶
📋 Answer

A read-write lock allows concurrent reads but exclusive writes. Multiple readers can hold the lock simultaneously. When a writer wants access, it waits for all readers to finish. This improves performance when reads are far more common than writes.

Reader-Writer problem: The challenge is preventing writer starvation — if readers keep arriving, the writer may never get access.

Q28. What is the difference between synchronous and asynchronous I/O? ▶
📋 Answer

Synchronous (blocking) I/O: The calling thread waits until the I/O completes. Simple but wastes CPU during waiting.

Asynchronous (non-blocking) I/O: The call returns immediately and the thread continues. The OS notifies when I/O completes (via callback, signal, or polling). More complex but allows concurrent operations — critical for high-performance servers, GUIs.

Examples: sync read() vs. async I/O using select(), epoll (Linux), IOCP (Windows).

Q29. What is the difference between concurrency and parallelism? ▶
📋 Answer

Concurrency: Multiple tasks making progress in overlapping time periods. Tasks may not run simultaneously — the CPU switches between them. About dealing with many things.

Parallelism: Multiple tasks executing at exactly the same time on different CPU cores. About doing many things simultaneously.

Concurrency can exist with a single core (via time-slicing); parallelism requires multiple cores.

Q30. What is the difference between symmetric and asymmetric multiprocessing? ▶
📋 Answer

SMP (Symmetric): All processors are equal, share memory, and can run any process. Most common (multi-core desktops/servers).

AMP (Asymmetric): Master-slave — one processor runs the OS, others run user tasks. Simpler but less efficient. Used in some embedded systems.

Memory Management & Virtual Memory

Q31. Explain the Memory Management Unit (MMU). How does it work? ▶
📋 Short Answer (30 sec)

The MMU is a hardware component that translates virtual addresses to physical addresses at runtime using page tables, enabling virtual memory and memory protection.

📝 Detailed Answer (2-3 min)

The MMU sits between the CPU and memory. When the CPU generates a virtual address, the MMU splits it into page number and offset, looks up the page number in the TLB or page table, retrieves the frame number, and combines it with the offset to form the physical address.

If the page is not in memory (page fault), the MMU raises an exception and the OS kernel handles loading the page from disk.

Q32. What is demand paging? How does it differ from anticipatory paging? ▶
📋 Short Answer (30 sec)

Demand paging loads pages into memory only when accessed. Anticipatory (prepaging) loads pages before they are needed, predicting future accesses.

📝 Detailed Answer (2-3 min)

Demand paging: When a process starts, no pages are loaded. Page faults occur as memory is accessed, and the OS loads required pages on demand. This minimizes initial load time and memory usage.

Prepaging: The OS predicts which pages will be needed and loads them in advance, reducing page faults but risking wasted I/O if predictions are wrong. Most modern OS primarily use demand paging.

Q33. Explain Belady's Anomaly in detail. ▶
📋 Short Answer (30 sec)

Belady's Anomaly is where increasing the number of page frames can paradoxically increase page faults. It only occurs with FIFO page replacement.

📝 Detailed Answer (2-3 min)

Reference string: 1,2,3,4,1,2,5,1,2,3,4,5. With 3 frames: 9 faults. With 4 frames: 10 faults. This violates the expectation that more memory reduces faults.

FIFO suffers from this because it replaces the oldest page regardless of usage. LRU and Optimal follow the stack property (page set with N frames is subset of N+1 frame set), guaranteeing no anomaly.

Q34. What is the difference between global and local page replacement? ▶
📋 Short Answer (30 sec)

Local replacement picks a victim from the faulting process's own frames. Global replacement can pick any frame from any process in the system.

📝 Detailed Answer (2-3 min)

Local: Each process has a fixed frame allocation. Isolates processes but may waste memory if some processes have idle frames while others need more.

Global: Dynamically adjusts frame allocation based on need. More efficient overall but a single misbehaving process can cause system-wide thrashing. Most modern OS use global replacement with working set monitoring.

Q35. Explain the concept of a working set and its significance. ▶
📋 Short Answer (30 sec)

The working set is the set of pages a process is actively using (its current locality). It defines the minimum number of frames a process needs to run without thrashing.

📝 Detailed Answer (2-3 min)

Working Set Model (Denning, 1968): W(t, Δ) = set of pages accessed in time interval [t-Δ, t]. If a process has fewer frames than its working set size, it will thrash.

The OS monitors working set sizes to decide which processes to keep in memory. When total working set size exceeds available RAM, the OS must suspend some processes to prevent system-wide thrashing.

Q36. Explain physical vs logical address and address binding. ▶
📋 Short Answer (30 sec)

Logical address is generated by CPU (virtual). Physical address is actual RAM location. Address binding maps logical to physical, happening at compile time, load time, or execution time.

📝 Detailed Answer (2-3 min)

Binding stages: Compile time (absolute code, must recompile if location changes), Load time (relocatable code, loader binds), Execution time (dynamic, requires MMU, enables virtual memory).

Modern OS use execution-time binding for virtual memory support and process relocation.

Q37. Explain Copy-on-Write (COW). ▶
📋 Short Answer (30 sec)

COW is an optimization where parent and child processes share pages after fork() instead of copying. Pages are duplicated only when either process writes to them, saving time and memory.

📝 Detailed Answer (2-3 min)

When fork() is called, the OS marks all pages as read-only and shares them. When either process writes, a protection fault triggers duplication just for that page. Benefits: faster fork(), memory savings when child calls exec() immediately, and shared code pages are never copied.

Q38. What is the Clock (Second Chance) page replacement algorithm? ▶
📋 Short Answer (30 sec)

The Clock algorithm uses a circular list of pages with reference bits. A hand pointer scans: if ref bit = 1 (recently used), clear and advance; if 0, replace that page. It approximates LRU efficiently.

📝 Detailed Answer (2-3 min)

Enhanced Clock (used in Linux) uses both reference and dirty bits with priority: (0,0) clean unused → (0,1) dirty unused → (1,0) clean used → (1,1) dirty used. This avoids writing clean pages to disk during replacement.

Q39. What are memory-mapped files? ▶
📋 Short Answer (30 sec)

Memory-mapped files (mmap) map a file into a process's virtual address space. The process accesses file content using regular memory operations instead of read/write syscalls.

📝 Detailed Answer (2-3 min)

Benefits: Simplified code (pointer operations), no syscall overhead per access, demand-paged loading, and shared mappings for IPC. Types: MAP_PRIVATE (changes not written back, COW) and MAP_SHARED (changes written to file).

Q40. What is the difference between stack and heap allocation? ▶

Stack: automatic, fast, LIFO, limited size (~1-8MB). Heap: manual (malloc/free), flexible, large, but slower with fragmentation risk.

File Systems & Storage

Q41. Explain hard links vs soft links (symlinks). ▶
📋 Short Answer (30 sec)

Hard link: additional directory entry pointing to the same inode. Soft link: special file storing a path to another file. Deleting the original breaks soft links but not hard links.

📝 Detailed Answer (2-3 min)

Key differences: Hard links can't cross filesystems or link directories. Soft links can. The inode's link count tracks hard links; data is freed when count reaches zero. Soft links become dangling when the target is deleted.

Q42. What is journaling in file systems? Why is it important? ▶
📋 Short Answer (30 sec)

Journaling records pending changes in a log before applying them. After a crash, the journal is replayed for consistency instead of running a full fsck scan.

📝 Detailed Answer (2-3 min)

Modes: Writeback (metadata only, least safe), Ordered (metadata with data written first, default in ext4), Data (both metadata and data, safest but slowest).

ext3, ext4, NTFS, APFS use journaling. ZFS and Btrfs use copy-on-write instead.

Q43. Compare ext2, ext3, and ext4 file systems. ▶
📋 Short Answer (30 sec)

ext2: no journaling. ext3: adds journaling, backward compatible. ext4: modern standard with extents, delayed allocation, and exabyte-scale support.

📝 Detailed Answer (2-3 min)

ext4 improvements: extents (efficient large file storage), delayed allocation (better performance, less fragmentation), max filesystem size 1 EB (vs 32 TB), max file size 16 TB (vs 2 TB), and online defragmentation.

Q44. What is an inode? What information does it store? ▶
📋 Short Answer (30 sec)

An inode is a disk data structure storing file metadata: permissions, ownership, timestamps, size, and block pointers. The filename is stored in the directory entry, not the inode.

📝 Detailed Answer (2-3 min)

Contents: mode/type, rwx permissions, UID/GID, atime/mtime/ctime, size, direct/indirect/double-indirect/triple-indirect block pointers, and link count. The inode number uniquely identifies the file. Directories map filenames to inode numbers.

Q46. What is the File Allocation Table (FAT)? ▶
📋 Short Answer (30 sec)

FAT is a filesystem using a table tracking disk block chains. Each directory entry points to the first block; the FAT entry links to the next, forming a linked list. Variants: FAT12, FAT16, FAT32, exFAT.

📝 Detailed Answer (2-3 min)

FAT32: max 2TB volume, 4GB file limit. exFAT: removes 4GB limit, designed for flash drives. Still widely used for USB drives and memory cards due to universal OS compatibility.

Q47. What is disk partitioning and why is it useful? ▶

Partitioning divides a physical disk into logical sections. Benefits: multi-boot, data isolation (corruption in one doesn't affect others), and faster fsck on smaller filesystems.

I/O Management & Device Drivers

Q48. Explain programmed I/O, interrupt-driven I/O, and DMA. ▶
📋 Short Answer (30 sec)

Programmed I/O: CPU polls device (wasteful). Interrupt-driven: device signals CPU when ready. DMA: controller transfers data directly between device and memory, CPU only sets up and handles completion.

📝 Detailed Answer (2-3 min)

This is the evolution of I/O techniques, each reducing CPU overhead. DMA is critical for high-speed devices (disks, graphics, network). Without DMA, the CPU would be constantly interrupted, severely limiting throughput.

Q49. What is an interrupt? Explain the handling process. ▶
📋 Short Answer (30 sec)

An interrupt is a hardware signal causing the CPU to pause, save state, and run a handler. After the handler completes, the CPU resumes the interrupted process.

📝 Detailed Answer (2-3 min)

Sequence: Device sends signal → CPU saves PC and registers → Looks up IVT for handler → Switches to kernel mode → ISR executes → Restores state → Resumes interrupted process.

Interrupts can be masked (delayed). The APIC manages interrupt prioritization.

Q50. What is the role of device drivers? ▶
📋 Short Answer (30 sec)

Device drivers are kernel modules providing a standard interface for OS-hardware communication. They translate generic I/O requests into device-specific commands.

📝 Detailed Answer (2-3 min)

Types: Character (stream: keyboard, mouse), Block (block-oriented: disks), Network (packet: NICs). In Linux, drivers can be built into the kernel or loaded as modules (LKMs). Driver bugs are a major crash cause since they run in kernel mode.

Q51. What is buffering in I/O? Why is it needed? ▶
📋 Short Answer (30 sec)

Buffering stores data temporarily during I/O to handle speed mismatches, data size differences, and provide copy semantics for consistency.

📝 Detailed Answer (2-3 min)

Three reasons: (1) Speed mismatch (fast CPU vs slow disk), (2) Size mismatch (network packets vs messages), (3) Copy semantics (application modifies buffer after write, kernel has original). Strategies: single, double (alternating), circular buffer.

Q52. What is blocking vs non-blocking I/O? ▶
📋 Short Answer (30 sec)

Blocking I/O suspends process until completion. Non-blocking returns immediately (with EAGAIN if no data). Non-blocking is essential for single-threaded servers and GUIs.

📝 Detailed Answer (2-3 min)

I/O multiplexing (select, poll, epoll) monitors multiple FDs and notifies when any are ready. This is the foundation of high-performance servers like nginx and Node.js. epoll scales better than select by using O(1) event notification.

Distributed Systems

Q53. What is the CAP theorem? ▶
📋 Short Answer (30 sec)

A distributed system can provide at most two of: Consistency (latest write), Availability (every request gets response), Partition Tolerance (works despite network failures). Since partitions are inevitable, systems choose CP or AP during partitions.

📝 Detailed Answer (2-3 min)

CP systems (ZooKeeper, etcd) reject some requests during partitions. AP systems (Cassandra, Dynamo) accept stale reads. During normal operation (no partition), systems can be both consistent and available.

Q54. What is a Remote Procedure Call (RPC)? ▶
📋 Short Answer (30 sec)

RPC allows calling a function on a remote server as if local. The framework handles serialization, transport, and errors transparently.

📝 Detailed Answer (2-3 min)

Flow: Client stub marshals args → sends over network → server stub unmarshals → calls function → result flows back. Challenges: latency, partial failures, endianness. Modern: gRPC (HTTP/2, Protobuf), Thrift, JSON-RPC.

Q55. Explain distributed mutual exclusion. ▶
📋 Short Answer (30 sec)

Ensures only one process across a network accesses a shared resource at a time. Approaches: centralized (coordinator), token-based (token ring), quorum-based (voting).

📝 Detailed Answer (2-3 min)

Centralized: 3 messages per CS, single point of failure. Token ring: 1 per pass, token loss risk. Ricart-Agrawala: 2(N-1) messages, needs multicast. Maekawa: O(√N) messages, can deadlock.

Q56. What is a distributed file system? Examples? ▶
📋 Short Answer (30 sec)

A DFS allows accessing remote files as if local. Examples: NFS (local network), AFS (enterprise), HDFS (big data), Ceph (cloud), GFS (Google).

📝 Detailed Answer (2-3 min)

Challenges: Network latency, cache consistency, concurrent updates, partial failures, security. HDFS uses write-once + large blocks for big data. NFS uses stateless protocol for simplicity.

Q57. Explain Lamport's Logical Clocks. ▶
📋 Short Answer (30 sec)

Lamport clocks assign monotonically increasing integers to events capturing the happened-before relationship. If A → B, then clock(A) < clock(B). Provides partial ordering.

📝 Detailed Answer (2-3 min)

Rules: (1) Each process has LC = 0. (2) Before each event: increment LC. (3) Send message with LC. (4) On receive: LC = max(LC, msg_ts) + 1. Vector clocks extend this with per-process arrays for concurrent event detection.

Q58. Compare strong, eventual, and causal consistency. ▶
📋 Short Answer (30 sec)

Strong: every read returns latest write (slow). Eventual: replicas converge over time (fast). Causal: causally related operations seen in same order by all processes.

📝 Detailed Answer (2-3 min)

Use cases: Strong = banking/databases (sync replication). Causal = collaborative editing. Eventual = DNS/social media (async replication). Weak = cache/analytics (fastest, no guarantees).

Q64. Explain the Linux Completely Fair Scheduler (CFS). ▶
📋 Short Answer (30 sec)

CFS is Linux's default scheduler (since 2.6.23). It uses a red-black tree keyed by vruntime to always pick the task that has run the least, ensuring fairness.

📝 Detailed Answer (2-3 min)

Q65. Explain priority inversion and priority inheritance. ▶
📋 Short Answer (30 sec)

Priority inversion: high-priority task blocked by low-priority task holding a lock, while medium task runs. Priority inheritance temporarily raises the low task's priority to fix this.

📝 Detailed Answer (2-3 min)

Mars Pathfinder bug (1997): Low task L holds mutex, high task H waits, medium task M preempts L. H is stuck. Fix: when H waits on L's mutex, L inherits H's priority temporarily, runs (preempting M), releases mutex, and priority returns to normal.

Q66. What is a spinlock? When to use it vs a mutex? ▶
📋 Short Answer (30 sec)

Spinlock: thread busy-waits until lock acquired. Use for very short waits (μs) in interrupt handlers where sleeping is unsafe. Use mutexes for longer waits where thread can sleep and context switch.

📝 Detailed Answer (2-3 min)

Spinlocks waste CPU while spinning but avoid context switch overhead. The Linux kernel uses spinlocks for short critical sections. Mutexes (like futex in Linux) put threads to sleep, using zero CPU but with context switch cost.

Q67. How does the Producer-Consumer problem work with multiple producers/consumers? ▶
📋 Short Answer (30 sec)

Three semaphores: mutex (buffer lock), empty (free slots), full (filled slots). Producer: wait(empty), wait(mutex), add, signal(mutex), signal(full). Consumer: wait(full), wait(mutex), remove, signal(mutex), signal(empty).

📝 Detailed Answer (2-3 min)

Critical ordering: Always wait(empty/full) before wait(mutex). Reversing them can deadlock if the buffer is full — producer holds mutex and waits on empty, but consumer can't enter because mutex is held.

FAQ 4: What is the difference between interrupt and exception? ▶
📝 Answer

Interrupt: An asynchronous event caused by hardware (keyboard key press, timer tick, disk I/O complete). The CPU saves its state and runs the interrupt handler.

Exception: A synchronous event caused by the running program itself (division by zero, page fault, invalid instruction). Also called a trap or fault.

Key difference: Interrupts come from outside the CPU (external devices). Exceptions are generated by the CPU itself due to program instructions.

🔗 Related Topics

Interrupt handler, ISR, trap vs fault vs abort

FAQ 5: What is the difference between logical address and physical address? ▶
📝 Answer

Logical (Virtual) Address: Address generated by the CPU during program execution. Each process has its own logical address space, starting from 0. Not directly usable by memory hardware.

Physical Address: Actual address in main memory (RAM). The memory unit uses physical addresses to access data.

The MMU (Memory Management Unit) translates logical addresses to physical addresses at runtime using page tables.

🔗 Related Topics

MMU, Address binding, Paging, TLB

FAQ 6: What is spooling? ▶
📝 Answer

Spooling (Simultaneous Peripheral Operations Online) uses disk as a large buffer for slow devices like printers. Multiple processes can write their output to the spool (disk) at the same time without waiting. A separate daemon reads from the spool and sends to the printer one job at a time.

Difference from buffering: Spooling overlaps I/O of one job with computation of another. Buffering overlaps input/output of a single job.

🔗 Related Topics

Buffering, Caching, I/O management
